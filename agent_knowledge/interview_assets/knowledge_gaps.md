# Knowledge Gaps And Boundaries

这个文件不是为了“补知识”，而是为了防止后续回答越界。

## 1. 当前材料强支撑的部分

- RAG 原理与优化
- 混合检索 / BM25 / Rerank / Query Rewrite
- Agent 工作流设计
- Tool Schema / ReAct / 多工具调用
- Function Calling SFT
- LoRA
- GRPO / DPO 基础理解
- Memory / MCP 的设计理解

## 2. 当前材料只中等支撑的部分

- Multi-Agent 的工程实现
- 长期记忆的生产级治理
- MCP 在真实 ToB 生产场景的稳定性和负载问题
- RLHF 在复杂任务中的系统化评测

这些内容回答时，建议使用：

- `我实践过`
- `我在课程项目里做过`
- `我对这套设计比较熟`

## 3. 当前材料弱支撑或不应硬补的部分

- OCR / ASR / 视频解析的真实线上方案
- 你们公司内部 35+ API 的真实治理细节
- 线上并发、QPS、容灾、成本
- 训练样本量、GPU、训练时间的精确数字
- 你所在团队的协作流程和组织方式

这些内容如果被追问，应优先：

1. 回到真实经历
2. 明确说“这部分以我实际项目为准”
3. 不要从课程资料中硬补

## 4. 最容易让回答变陌生的几类扩展

- 一上来就讲太多论文术语
- 把 Demo 场景说成公司真实场景
- 用过于学术化的 RL 公式替代工程语言
- 引入你没明确学过的热门框架当主线答案

## 5. 遇到不会或材料不足时的推荐回答方式

### 模板 1：工作经历优先

`这部分我课程里有相关设计思路，但如果严格对应到我实际项目，我会更倾向于这样回答……`

### 模板 2：课程实践边界

`这个点我系统学过，也做过 Demo 级实践，但如果说生产级落地，我会把它定位成探索和预研。`

### 模板 3：保守表达

`我对这套方案的原理和实现路径比较熟，不过具体到线上规模和成本，我不想脱离真实经历去夸大。`
