# Persistent User Constraints

这个文件用于固化用户已经明确提出过、后续不希望反复重说的约束。

默认规则：

- 只要任务和简历、模拟面试、项目复盘、面试问答有关，就先读这个文件。
- 除非用户在后续消息里明确覆盖，否则这里的约束长期有效。
- 如果后续要求发生变化，应优先更新这个文件，再继续生成内容。

## 当前长期约束

### 1. 目标岗位

- 目标岗位：`Agent 应用开发工程师`

### 2. 知识边界

- 回答必须优先贴合你的简历主项目。
- 回答必须尽量落在你已经学过、看过、做过或实践过的知识范围内。
- 优先使用 `agent_knowledge/`、`ready/`、`agent资料/` 中已有材料。
- 不要为了显得高级，主动引入你没学过或资料里没重点覆盖的陌生方案。

### 3. 表述强度

- 简历明确写出且有材料支撑的内容，才用 `做过`、`负责`、`落地过` 这类强表述。
- 课程项目、Demo、PoC 支撑的内容，使用 `实践过`、`学过`、`搭过 Demo`。
- 不要把课程 Demo 指标、演示效果、实验结果说成真实工作项目的线上成果。
- 不要编造模型版本、QPS、样本规模、训练时长、机器配置等未确认信息。

### 4. 生成内容的语气

- 回答语气必须贴近真实面试场景。
- 生成的参考回答应当可以直接读给面试官听。
- 优先使用口语化但专业的表达，而不是文档腔、论文腔或宣传腔。
- 先正面回答问题，再补方法、权衡和结果。

### 5. 模拟面试的形式

- 面试题要贴合简历，而不是只贴合课程。
- 不只生成散题，优先生成带追问链的拟真面试。
- 回答要尽量像真实候选人临场回答，而不是知识点堆砌。

### 6. 输出目录约束

- 后续生成的交付内容默认放在 `D:\code\agent_data\output\`。
- 包括但不限于：模拟面试题、模拟面试问答、STAR 版本、项目复盘稿、JD 定制问答。

### 7. 输出格式偏好

- 模拟面试问答默认使用 `问题 1`、`问题 2`、`问题 3` 这样的编号格式。
- 不使用 `第 1 轮`、`第 2 轮` 这类标题格式，除非用户后续明确要求。

### 8. mock 文件命名约束

- 只有“按 200 题分卷的正式模拟面试题库”使用 `mock_` 前缀。
- 在题量未满 `200` 之前，`output/` 下默认只保留一个 `mock_*.md` 题库文件。
- 参考回答稿、连续追问样例、索引说明这类辅助文件，不使用 `mock_` 前缀，统一使用 `interview_` 前缀。

### 9. 工作方式约束

- 后续不要依赖“记住这次对话”的临时上下文。
- 应把本文件视为长期有效的本地约束源。
- 如果进入长对话或新一轮任务，先回到本文件校准输出方式。

## 后续维护方式

如果用户又补充了新的稳定偏好，例如：

- 更偏一面风格还是二面深挖风格
- 更偏口语还是更偏技术
- 某些不能说的真实边界
- 某些特别希望强化的项目点

应直接追加到本文件，而不是只保留在聊天上下文里。
