## 1. 使用说明

- 这份文档不是知识百科，而是按你当前简历内容生成的面试训练稿。
- 每道题都包含三部分：`模拟问题`、`优质回答`、`资料库内容`。

- 整体答题顺序建议固定为：`业务背景 -> 问题定义 -> 系统方案 -> 优化点 -> 指标结果 -> 我的职责`。

- 刷新说明：搜狐经历统一按 `2024-10 至今`，核心项目统一按 `2025-06 至今`。
- 刷新说明：面试表达里优先使用 `独立测试集 / 独立评测集`，不优先使用 `held-out`。

## 2. 统一主叙事

你整段项目经历最稳的讲法可以固定成下面这段：

我原来做的是一个基于 `Django4 + DRF` 的游戏测试管理平台，平台里已经有版本规划、测试任务、测试用例、缺陷、测试进度、TR 报告和附件等业务模块。后面我做的不是重新起一个 AI Demo，而是把这个平台升级成一个面向测试设计、执行编排和发布风险判断的 Agent 应用平台。一方面，我用 `DataAgent + RAG` 把规划文档、脑图、依赖图、历史用例、缺陷趋势、评审意见、周报和多模态附件处理成可检索、可引用的私有知识；另一方面，我把平台内部 `35+` 个高频 Django API 封装成 Tool Schema，让 Agent 不只是回答问题，而是能真实完成测试用例生成、执行建议、风险诊断和周报汇总。因为多工具、多参数、多步链路下单靠 Prompt 不稳定，我又主导了 Function Calling 的 `SFT` 数据工程，并重点参与离线评测、迭代闭环和 `GRPO` 方向优化，最终把这个项目从传统测试后端升级成了一个具备知识检索、工具调用、工作流编排和质量决策能力的智能测试系统。

### 题 1：请你用 1 到 2 分钟介绍一下你这个项目

`模拟问题`

请你完整介绍一下“基于 RAG + Function Calling 的游戏智能测试 Agent 平台”这个项目。

`优质回答`

这个项目本质上是对原有游戏测试管理平台的一次 Agent 化升级。原平台已经有版本规划、任务中心、测试用例、缺陷、测试进度、TR 报告和附件等模块，但测试设计、执行编排和发布风险判断这几类高价值工作仍然比较依赖资深测试人员手工完成，知识分散、用例编写耗时、风险评审也比较依赖经验。我做的核心工作有三块。第一块是知识层，我设计了 `DataAgent + RAG` 两阶段链路，把规划文档、脑图、依赖图、历史用例、缺陷趋势、评审意见、周报和截图视频等多源数据做解析、切分、元数据建模和离线入库，构建测试场景私有知识库。第二块是工具层，我把平台内 `35+` 个高频 Django API 抽象成 Tool Schema，基于 `ReAct + Plan/Act/Reflect + Memory` 设计测试设计 Agent、执行编排 Agent 和质量诊断 Agent，让系统能完成检索知识、调用工具、生成建议、结果回写的完整闭环。第三块是模型稳定性优化，因为工具多、参数复杂、链路是多步的，单靠 Prompt + RAG 不稳定，所以我主导了基于 `Qwen2.5-7B + LoRA` 的 Function Calling SFT 数据工程，并重点参与离线评测和迭代闭环，同时进一步探索 `GRPO` 优化多工具决策。最终离线检索上 `Recall@10` 从 `67%` 提升到 `89%`，`MRR` 从 `0.49` 提升到 `0.77`，工具选择准确率从 `79%` 提升到 `92%`，参数 `EM` 从 `71%` 提升到 `86%`，测试用例初稿生成效率提升约 `70%`。

### 题 3：为什么一个测试平台天然适合做 Agent 升级

`模拟问题`

为什么你们这个项目适合做 Agent，而不是只是做一个普通问答机器人？

`优质回答`

因为这个场景天然就是多步任务，不是单轮问答。测试负责人真正关心的问题通常是“根据版本规划生成任务树和首版测试用例”“结合历史缺陷和当前进度判断是否可提测”“根据 TR 报告和缺陷趋势生成发布风险摘要”。这类问题同时依赖企业私有知识、平台内部工具能力和多步决策逻辑。外部搜索拿不到版本规划、历史用例、缺陷趋势这些内部数据；如果只做问答，系统只能给建议，不能真正去取数、分析、生成报告和回写结果；如果不做规划与反思，复杂链路又很容易在中间一步失败。所以这个场景非常适合用 `RAG + Tool Use + Workflow` 的方式做 Agent 化升级。

### 题 5：DataAgent + RAG 两阶段链路是怎么设计的

`模拟问题`

你简历里写了 DataAgent + RAG 两阶段知识处理链路，具体是什么意思？

`优质回答`

我把它拆成两个阶段理解。第一阶段是 DataAgent，解决的是“原始数据不能直接被模型稳定消费”的问题。因为平台里既有结构化数据，比如任务、缺陷、进度、报表，也有半结构化和非结构化数据，比如测试用例文本、评审意见、PDF、Word、Excel、截图、视频、脑图和依赖图，所以要先做文档解析、OCR/ASR、节点关系抽取、chunk 切分和 metadata 建模，把这些内容变成可检索、可过滤、可回溯的知识对象。第二阶段是 RAG，解决的是“怎么把对的知识在对的场景里召回出来”的问题。这里我做的是向量检索、关键词检索、重排、Query Rewrite、元数据过滤和多轮上下文继承，让测试设计、执行建议和质量诊断类问题都能先检索到高相关上下文，再去生成和调用工具。

### 题 6：为什么选 BGE-M3 + BM25/Elasticsearch + Cross-Encoder Rerank

`模拟问题`

你们为什么不是只做向量检索，而是要混合检索加重排？

`优质回答`

因为测试平台里的查询天然同时包含语义匹配和精确匹配需求。比如版本号、模块名、任务名、缺陷类型、TR 报告编号这类信息，关键词检索通常更稳；但像“这轮回归可能遗漏哪些风险点”“基于历史相似版本生成测试建议”这种问题，语义检索更有价值。所以单做向量检索容易漏掉精确实体，单做 BM25 又容易漏掉语义相近内容。我的做法是先用 `BGE-M3` 做语义召回，再结合 `BM25/Elasticsearch` 做关键词补召回，最后用 `Cross-Encoder Rerank` 二次排序，把真正和当前问题强相关的文档片段排到前面。这个组合的意义是兼顾召回覆盖率和排序精度，也是我们 `Recall@10` 和 `MRR` 能明显提升的关键原因。

### 题 7：你们怎么做 Query Rewrite、元数据过滤和评估指标

`模拟问题`

你提到 Query Rewrite、元数据过滤和 Recall/MRR，这几块是怎么结合的？

`优质回答`

Query Rewrite 主要解决用户问题表达和平台知识组织方式不一致的问题。真实用户经常是口语化提问，比如“这轮还有哪些风险”“帮我看下这个版本能不能提测”，这类问题直接检索命中率不高，所以要把问题改写成带有项目、版本、模块、任务、时间范围等约束的内部表达。元数据过滤则进一步缩小检索范围，比如限定 `game_project_id`、`version_planning_id`、`milestone_id`、`task_id` 或时间窗口，避免把其他版本、其他模块的内容误召回。评估上我会分层看，检索层重点看 `Recall@10` 和 `MRR`，前者看候选有没有把正确文档召回来，后者看正确结果排得是否靠前；如果往上再看，会结合多轮一致性、端到端任务成功率和引用可信度。这样才能知道问题到底出在召回、排序还是生成阶段。

### 题 8：为什么要把 35+ 个 Django API 封装成 Tool Schema

`优质回答`

如果不做工具化，模型最多是个咨询助手，不能真正进入业务闭环。这个平台原来就已经有大量真实可用的后端 API，比如任务创建与更新、执行建议、缺陷统计、质量概览、周报和 TR 报表相关接口，这些都是现成的业务能力。把它们封装成 Tool Schema 之后，Agent 才能从“回答问题”升级成“查数据、调能力、生成结果、必要时回写平台”的执行系统。Tool Schema 的核心价值有三个，一是统一工具的输入输出格式，便于模型学习和调度；二是补足参数约束、用途说明、错误处理和读写权限分级；三是让平台已有资产能够被 Agent 直接复用，而不是新造一套脱离业务的假工具。

### 题 9：你们的 Agent 工作流是怎么落地的

`模拟问题`

简历里写了 ReAct、Plan/Act/Reflect、Memory，这些在项目里是怎么落地的？

`优质回答`

我在项目里不是把这些词当概念，而是把它们放进具体流程里。Plan 阶段主要负责理解用户意图和拆分子任务，比如先判断当前问题属于测试设计、执行编排还是质量诊断。Act 阶段负责两类动作，一类是检索知识，一类是调用工具。Reflect 阶段主要做结果校验，比如当前信息够不够、工具顺序是否合理、是否缺少关键上下文、引用依据是否充分。Memory 则负责保留项目上下文、版本上下文和多轮对话历史，避免每次都从零开始。落到业务上，我们拆成了测试设计 Agent、执行编排 Agent 和质量诊断 Agent，不是为了追求多智能体名词，而是因为三类场景依赖的知识、工具和输出结构都不同，分开建更容易控流程、控质量。

### 题 10：多工具、多步骤下怎么控制安全性和稳定性

`模拟问题`

既然 Agent 能调很多内部接口，你们怎么避免乱调用，尤其是写操作风险？

`优质回答`

这类平台不能把 Agent 当成完全无约束的执行器，尤其是涉及写接口时。我的做法是先做工具分级，把只读查询工具和有副作用的写工具区分开；再在 Tool Schema 层补充参数约束、幂等要求和错误处理；高风险写操作要么加确认机制，要么限制在白名单场景；结果输出必须尽量带来源引用，方便审计和回放；如果是多步链路，还会在 Reflect 阶段做一次结果审校，避免顺序错、参数错或者上下文不充分时直接写入。简单说，这里不是只追求“能调”，更要保证“调得稳、调得可追溯、调错了能定位”。

### 题 11：为什么选 Qwen2.5-7B + LoRA，而不是全量微调

`模拟问题`

为什么你们工具调用模型选的是 `Qwen2.5-7B + LoRA`？为什么不是更大模型或者全量微调？

`优质回答`

这个选择更多是工程约束和任务匹配的结果。我们的目标不是训练一个通用聊天模型，而是提升 Function Calling 场景下的工具选择、参数抽取和多步链路稳定性，所以更关注结构化输出、成本和迭代效率。`Qwen2.5-7B` 在中文理解、工具调用适配和指令跟随方面已经比较适合这类任务，而 `LoRA` 的好处是训练和迭代成本低，适合我们快速做 badcase 驱动优化。如果上来就全量微调，一方面资源成本高，另一方面对这种相对垂直、动作空间明确的任务，性价比并不一定更好。我的理解是，只有当任务复杂度、数据规模和资源预算都支持时，全量微调才更有必要；在我们的阶段，`7B + LoRA` 是更务实的工程选型。

### 题 14：你们怎么评估 SFT 的效果

`模拟问题`

`优质回答`

## 题 156：为什么 metadata 过滤不能省

## 三、指标追到公式题

## 题 161：Recall@10 公式怎么讲

`模拟问题`

如果我让你直接写 Recall@10 的定义，你怎么说？

`参考回答`

最稳的讲法是：对每个 query，看前 10 个检索结果里是否命中人工标注的正确证据集合。单条样本如果命中，就记 1，否则记 0；所有样本求平均，就是 Recall@10。更严格一点的表述可以说：

`Recall@10 = 命中正确证据的 query 数 / query 总数`

这里前提是先定义好每个 query 的“正确证据”。

---

## 题 162：MRR 公式怎么讲

`模拟问题`

MRR 怎么算？别只说“看排序靠前不靠前”。

`参考回答`

MRR 看的是第一条正确结果出现的位置。对每个 query，找到第一个命中的正确证据排在第几位，记为 `rank_i`，这条样本的分数就是 `1 / rank_i`；如果前面都没命中，就记 0。所有 query 的这个值求平均，就是 MRR。公式可以讲成：

`MRR = (1 / N) * Σ (1 / rank_i)`

其中 `N` 是 query 数，`rank_i` 是第一个正确结果的位置。

---

## 题 163：工具选择准确率公式怎么讲

`模拟问题`

工具选择准确率怎么定义最清楚？

`参考回答`

单步场景下最简单：

`Tool Selection Accuracy = 选对目标工具的样本数 / 总样本数`

如果是多步场景，可以进一步拆成单步工具准确率和链路级工具序列正确率。面试里最稳的是先把单步口径讲清楚，再补一句：多步任务我们还会单独看链路级成功率。

---

## 题 164：参数 EM 公式怎么讲

`模拟问题`

参数 EM 你怎么定义？

`参考回答`

参数 `EM` 本质上是 Exact Match。也就是模型输出的结构化参数在归一化后，与标准答案完全一致才记 1，否则记 0。所有样本求平均，就是参数 EM。可以讲成：

`Parameter EM = 完全匹配样本数 / 总样本数`

这里“完全匹配”的前提是做过归一化，比如同义字段值、大小写、枚举写法先统一。

---

## 题 165：Overall Call Accuracy 怎么讲

`模拟问题`

整体调用准确率和工具准确率有什么区别？

`参考回答`

Overall Call Accuracy 更严格，它要求不仅工具选对，参数也对，而且最终调用链路满足成功条件。最简单可以讲成：

`Overall Call Accuracy = 调用整体成功样本数 / 总样本数`

它比工具选择准确率更接近真实业务，因为它把动作完整性也算进来了。

---

## 题 166：多步任务成功率怎么讲

`模拟问题`

你简历里有多步任务成功率，这个公式怎么定义？

`参考回答`

最稳的定义是：在一组多步任务评测样本里，最终完成目标任务且关键步骤满足预期的样本占比。可以讲成：

`Multi-step Task Success Rate = 成功完成目标任务的样本数 / 多步任务总样本数`

关键是先定义“成功”标准，比如该查的知识查到了、该调的工具调对了、最终产物达到可评审要求。

---

## 题 167：无效工具调用率怎么讲

`模拟问题`

无效工具调用率怎么定义更合理？

`参考回答`

它一般看的是所有工具调用里，有多少次调用是冗余、错误、无意义或不该发生的。可以讲成：

`Invalid Tool Call Rate = 无效工具调用次数 / 工具调用总次数`

其中“无效”要提前定义清楚，比如重复调用同一查询、错工具、错顺序、参数缺失导致必然失败等。

---

## 题 168：测试用例初稿生成效率提升 70% 怎么算

`模拟问题`

这个 70% 是怎么算出来的？

`参考回答`

最稳的说法是拿人工流程和 Agent 辅助流程做对比，看从输入规划材料到产出“可评审首版测试用例草稿”的平均或中位耗时。公式可以讲成：

`Efficiency Gain = (人工耗时 - Agent 辅助耗时) / 人工耗时`

如果这个值约等于 `70%`，就可以说效率提升约 70%。这里测的是首版草稿效率，不是最终全流程工时。

---

## 题 169：管理层风险摘要 10 分钟级，怎么定义口径

`模拟问题`

这个业务指标怎么避免被理解错？

`参考回答`

要明确它指的是“首版可评审风险摘要草稿”的生成时间，而不是整个管理决策闭环。测量方式还是人工流程和 Agent 辅助流程对比，只不过对象换成风险摘要任务。这样讲法更稳，不会让人误解成“10 分钟就决定是否上线”。

---

## 题 170：为什么你不报 BLEU、ROUGE 这类指标

`模拟问题`

很多生成任务都会报 BLEU、ROUGE，你们为什么不报？

`参考回答`

因为这个项目的关键不是“生成文本像不像标准答案”，而是“动作对不对、证据对不对、任务有没有完成”。所以我们优先看 Recall、MRR、工具准确率、参数 EM、多步任务成功率和业务效率指标。这些指标和实际场景的贴合度更高。BLEU、ROUGE 这类指标对 Function Calling 和 Agent 工作流帮助很有限。

---

## 题 171：如果面试官要你现场口头写一个完整评测公式组合，你怎么总结

`模拟问题`

你用一句比较系统的话，把你们的评测体系说完。

`参考回答`

我会说，我们把评测拆成三层：检索层用 Recall@10 和 MRR，工具层用工具选择准确率、参数 EM 和 Overall Call Accuracy，工作流层用多步任务成功率和无效工具调用率，业务层再看测试用例初稿和风险摘要的产出效率。这样每一层都有对应公式和口径，不会把所有结果混成一个模糊分数。

## 1. 架构与系统设计

### 题 21：如果让你画这个系统的总体架构，你会怎么拆层

`模拟问题`

如果不用讲实现细节，只画系统架构，你会怎么拆？

`参考回答`

我会把这个系统拆成五层。第一层是数据接入层，负责接版本规划文档、脑图、依赖图、历史用例、缺陷、周报、TR 报告和截图视频等多源数据。第二层是 DataAgent 层，负责解析、OCR/ASR、结构化增强、chunk 切分和 metadata 建模。第三层是知识与工具层，知识层是向量检索、关键词检索和重排组成的 RAG，工具层是平台原有 Django API 抽象出来的 Tool Schema。第四层是 Agent Orchestrator，负责意图识别、任务拆解、检索、工具调用、反思纠错和结果回写。第五层是训练与评测层，主要包含 Function Calling SFT、GRPO 探索、离线评测和 badcase 回放。这样拆的好处是业务数据、知识能力、执行能力和模型优化彼此解耦，后续每层都能独立迭代。

### 题 22：为什么不把知识检索和工具调用混在一个 Agent 里

`模拟问题`

为什么你们要强调知识层和工具层分开？一个大 Agent 不就够了吗？

`参考回答`

如果把知识检索和工具调用完全混在一起，调试和评估都会非常困难。因为一旦输出不对，你很难判断问题到底出在检索没召回、排序有噪声、工具选错、参数抽错，还是工作流顺序错了。把知识层和工具层分开之后，我可以先单独评检索质量，比如 Recall@10 和 MRR；再单独评工具选择和参数抽取；最后再评端到端任务成功率。对线上系统来说，这种解耦也更利于灰度和回滚，比如检索链路可以单独替换 rerank 策略，而工具调用层不需要跟着一起改。

### 题 23：为什么测试设计、执行编排、质量诊断要拆成多个 Agent

`模拟问题`

为什么不做一个统一的大助手，而是拆成多个 Agent？

`参考回答`

因为这三类任务的输入上下文、工具集合和输出结构差异都很大。测试设计更依赖规划文档、脑图、历史用例和缺陷知识，输出是任务树和用例初稿；执行编排更依赖任务进度、执行状态和阻塞信息，输出是优先级、回归范围和执行建议；质量诊断则更依赖缺陷趋势、质量检测、TR 报告和周报，输出是风险等级、问题清单和管理摘要。如果都揉成一个大助手，Prompt 会越来越复杂，工具选择空间也会过大，导致稳定性下降。拆开之后，每个 Agent 都能围绕自己那类任务做更清晰的检索、工具和输出设计。

### 题 24：如果未来要扩展成多智能体协作系统，你会怎么做

`模拟问题`

你现在是多角色工作流，如果继续往多智能体系统演进，你会怎么拆？

`参考回答`

我会先保留当前按业务职责拆分的方式，再加一个更强的 Orchestrator。Orchestrator 负责判断用户需求属于测试设计、执行分析还是质量决策，并决定是否需要跨 Agent 协作。比如一个复杂问题可能先让测试设计 Agent 产出用例覆盖建议，再让质量诊断 Agent 基于缺陷趋势和质量数据做风险评审。这个时候关键不是堆更多 Agent，而是要定义清楚：任务边界、状态传递、上下文共享方式、错误回退策略和最终汇总机制。否则多智能体只会把单体 Agent 的问题放大。

## 2. 数据与知识治理

### 题 25：你们接入的数据源很多，怎么保证数据质量

`模拟问题`

规划文档、截图、视频、历史用例、缺陷这些数据来源很杂，你们怎么做数据治理？

`参考回答`

我的思路是把数据治理前置到 DataAgent，而不是把脏数据直接塞给 RAG。具体会做四件事。第一是统一数据对象模型，把文档、用例、缺陷、报告都抽象成可检索知识对象，而不是保留纯原始文件。第二是做结构化增强，比如补项目、版本、任务、模块、时间范围这些 metadata。第三是做质量过滤，比如 OCR 质量差的片段、不完整文本、重复上传内容要过滤或降权。第四是做版本化管理，避免历史旧数据覆盖当前版本上下文。只有这样，后面的检索和生成才不会建立在混乱数据上。

### 题 26：metadata 是怎么设计的，为什么重要

`模拟问题`

你反复提 metadata 建模，它到底解决什么问题？

`参考回答`

metadata 的价值在企业场景里非常大，因为很多错误不是“没召回”，而是“召回了不该召回的东西”。比如同一个模块在不同版本、不同里程碑下的风险完全不一样，如果没有 metadata 过滤，很容易把历史旧版本内容混进来。我的设计重点会放在几类字段：项目、版本、里程碑、任务、模块、文档类型、时间范围、风险等级和来源类型。这样做之后，Query Rewrite 生成的约束条件就能和 metadata filter 对齐，检索空间会明显更可控。

### 题 27：截图和视频这类多模态素材，你们是怎么用起来的

`模拟问题`

你简历里写了截图、视频和附件素材，这类数据在系统里具体怎么进入知识链路？

`参考回答`

我的稳妥说法是，这类数据不是直接作为原始媒体给模型看，而是先做可检索化处理。截图主要通过 OCR 提取文字信息，并保留和任务、模块、版本相关的上下文；视频可以做帧抽样，再结合 OCR 或 ASR 提取关键信息。最终进入知识库的不是“视频文件本身”，而是从中抽出来的文本、时间点、截图说明和来源引用。这样做的目的，是把多模态素材转成能被检索和引用的知识对象，而不是让模型在长链路里直接消费原始多媒体。

### 题 28：如果文档不断更新，知识库怎么增量维护

`模拟问题`

版本规划和测试资料是一直在变的，知识库怎么避免过期？

`参考回答`

这种场景里知识库不能只做一次性离线构建，必须有增量更新机制。比较稳的设计是按数据源和业务对象做增量同步，比如文档更新、缺陷新增、周报生成、任务状态变化时触发对应对象的重解析和重建索引。同时要保留版本粒度的 metadata，让当前版本和历史版本能共存而不是互相覆盖。这样当用户问当前版本问题时优先命中当前上下文，问历史复盘问题时又能回溯到旧资料。

## 3. 工具设计与 Function Calling

### 题 29：Tool Schema 设计时你最关注哪些字段

`模拟问题`

如果让你定义一个 Tool Schema，你会重点设计什么？

`参考回答`

我会重点看五个方面。第一是工具用途描述，要让模型知道什么时候该调它。第二是参数定义，包括字段名、类型、是否必填、枚举范围和默认值。第三是输入输出格式统一，尤其是结构化返回不能太随意，否则后续 Agent 很难整合。第四是错误处理和幂等说明，尤其是写工具必须明确失败如何返回。第五是权限等级，读工具和写工具不能混为一谈。其实 Tool Schema 的本质不是“写个 JSON”，而是把业务接口抽象成模型可以稳定理解和调用的动作空间。

### 题 30：35+ 个工具会不会太多，模型怎么不混淆

`模拟问题`

工具这么多，模型怎么避免混淆？你怎么控制工具爆炸？

`参考回答`

工具数量多本身不是问题，真正的问题是工具边界模糊。我的处理思路是：第一，按领域分组，比如用例、任务、缺陷、质量、报表、附件，避免所有工具平铺到一个列表里。第二，工具描述里要强调使用场景，而不只是接口名字。第三，尽量避免多个工具做高度重复的事，否则模型会学不清边界。第四，能合并的读接口适当合并，能拆开的高风险写接口一定拆开。这样模型面对的不是 35 个杂乱动作，而是多个边界清晰的工具簇。

### 题 31：如果模型选对了工具但参数总是漏，你会先改什么

`模拟问题`

线上发现模型工具选得还行，但参数经常漏抽，你优先优化哪里？

`参考回答`

我会按成本从低到高排查。先看 Tool Schema 参数设计是不是对模型不友好，比如字段名不清晰、必填约束没表达清楚、多个参数语义相近。再看 Prompt 和 few-shot 样本是不是足够覆盖歧义表达。接着看数据集里有没有足够多“相同工具、不同参数形态”的样本，以及缺失参数场景的负样本。只有这些都做了还不够，我才会进一步依赖 SFT 或对齐优化。因为参数漏抽很多时候不一定是模型能力不够，也可能是工具定义和数据构造方式有问题。

### 题 32：对于写工具，你们如何避免误操作

`模拟问题`

如果 Agent 有创建任务、更新状态之类的写工具，怎么避免误写？

`参考回答`

写工具一定不能和查询工具同等对待。我的做法会是：先做权限分级，默认优先只放开读工具；必须放开的写工具要加参数校验和业务约束；高风险操作最好加确认机制或白名单；关键写操作保留调用日志、上下文和结果审计，方便回溯；如果是多步链路，还要在 Reflect 阶段做一次结果复核。简单说，写工具的目标不是让 Agent 更强，而是要在“可用”和“可控”之间找到平衡。

## 4. SFT、GRPO 与坏例处理

### 题 33：你们的 badcase 是怎么沉淀和回流的

`模拟问题`

Function Calling 做稳定之后，badcase 闭环怎么做？

`参考回答`

比较稳的闭环应该有三步。第一步是记录问题，包括用户输入、检索结果、模型中间决策、最终工具调用和失败原因。第二步是归因，把 badcase 分成检索问题、工具选择问题、参数抽取问题、链路编排问题和输出总结问题。第三步是回流，能用规则和工具定义修的先修规则，能用数据补齐的回流到 SFT 数据集，属于多步策略问题的再看是否进入 GRPO 奖励优化。这样闭环下来，badcase 不是堆日志，而是能形成持续迭代的数据资产。

### 题 34：如果训练集和评测集泄漏，你怎么发现

`模拟问题`

你怎么确保独立测试集 / 评测集真的有意义？如果泄漏了怎么办？

`参考回答`

这类问题我会从样本切分方式和线上表现两边看。切分上最好按场景或时间做隔离，而不是只随机切分，因为随机切分很容易把同一类样本的轻微改写分到训练和测试两边。现象上如果离线指标非常高，但线上遇到稍微变形的输入就明显掉效果，那通常就要怀疑泄漏或过拟合。更稳的方式是做模板外样本、歧义表达和跨版本场景评测，看看模型是不是只记住了数据形式，而不是真学会了工具调用。

### 题 35：GRPO 适合解决什么问题，不适合解决什么问题

`模拟问题`

你们做了 GRPO，那它最适合优化什么？什么问题不该交给它？

`参考回答`

GRPO 更适合优化多步工作流里的相对决策质量，比如工具链顺序、冗余调用控制、任务完成度和引用可信度。它适合解决“多个候选链路哪个更优”的问题，不适合拿来替代基础的数据清洗、Tool Schema 设计或者明显的参数定义错误。换句话说，如果一个问题本来是 schema 设计不清、训练数据不够或者检索链路有脏数据，用 GRPO 去补往往性价比不高。它更像是在基础能力已经比较稳的前提下，进一步把多步任务做得更优。

### 题 36：为什么不直接追求端到端一个指标

`模拟问题`

既然最终看业务结果，为什么还要拆这么多层指标？

`参考回答`

因为端到端指标只能告诉你“坏了”，但不能告诉你“坏在哪”。一个多工具 Agent 系统至少有三层误差来源：知识召回、动作决策和结果整合。如果只看最终业务指标，比如任务成功率或者人工采纳率，一旦效果下降就很难定位到底是检索没召回、工具选错、参数漏抽，还是工作流顺序出了问题。所以我会坚持分层评估：检索层、工具层、工作流层和业务层，这样每一层都有自己的优化抓手。

## 5. 稳定性、上线与灰度

### 题 37：如果让你把这个系统灰度上线，你会怎么做

`模拟问题`

这种 Agent 系统风险不低，如果灰度上线你怎么设计？

`参考回答`

我会优先从低风险场景开始，而不是一上来就放开全量自动执行。第一阶段先做只读辅助，比如风险摘要、执行建议、知识问答这类不直接改业务数据的能力。第二阶段再灰度到低风险写操作，比如生成用例草稿，但仍然要求人工确认。第三阶段才考虑更深的自动化闭环。整个过程中要配套做日志、采纳率、回退率、人工改写率和 badcase 回放。这样做的目标不是追求最快上线，而是先把稳定性和可解释性跑出来。

### 题 38：线上最可能翻车的环节是什么

`模拟问题`

你觉得这个系统线上最容易翻车的点在哪？

`参考回答`

我会认为最容易翻车的不是模型“答错一句话”，而是多步链路里的局部错误被放大。比如检索把旧版本知识召回进来、工具选对但参数漏了一个关键字段、写工具没有被正确拦截、结果引用不充分却被当成可靠结论输出。这些问题单看每一步可能都不致命，但串起来就会直接影响业务信任。所以线上最要命的不是某个模型指标低，而是链路失真后缺少拦截和回退机制。

### 题 39：如果延迟太高，你会先优化哪里

`模拟问题`

RAG、重排、工具调用、多步反思都很耗时，延迟高了怎么优化？

`参考回答`

我会先按链路拆耗时，再看哪些是高价值、哪些是可降级的。通常优先排查的是重排开销、无效检索次数和多余工具调用，因为它们往往既耗时又可能引入噪声。其次是按场景做分级，不是所有问题都需要最完整的链路，比如简单问答可能不需要多轮反思或复杂 rerank。再往后可以做缓存、并行化和轻重模型分层，比如简单任务用轻模型路由，复杂任务再进入完整链路。核心思想是先减少不必要步骤，而不是盲目堆机器。

### 题 40：如果业务方质疑 AI 结果不可靠，你怎么建立信任

`模拟问题`

测试和质量场景对准确性要求很高，你怎么让业务方敢用？

`参考回答`

我会从三个方面建立信任。第一是可解释性，输出尽量带引用，让业务方知道结论来自哪些规划文档、缺陷趋势或报告片段。第二是分级使用，先从辅助决策类场景切入，而不是直接替代人工做最终决策。第三是稳定性度量，用离线评测、灰度效果、采纳率和 badcase 回放去证明系统不是“偶尔答得好”，而是整体可控。对这种场景来说，信任不是靠一句“模型很强”建立的，而是靠可追溯和可回滚。

## 题 101：你说 Recall@10 提升了，那“正确证据”到底怎么定义

`主问题`

你说 Recall@10 从 67% 提升到 89%，那你们怎么定义什么叫“召回对了”？

`继续追问`

如果一个 query 能对应多个文档片段，你是按哪个算正确？

`参考回答`

这个指标前提一定是先有标注口径。我的理解是，先基于业务问题构造 `query-evidence` 对，人工标注该问题最关键的支撑证据片段或证据集合。评估时看前 10 个候选里有没有命中这些人工标注的证据。如果一个 query 对应多个有效证据，我会把它按证据集合处理，只要前 10 命中任一关键证据，就算 Recall 命中；如果要更严格，也可以把核心证据和辅助证据分层标注。重点是先统一口径，不然 Recall 数字本身没有意义。

## 题 102：一个 query 对应多个正确答案时，MRR 怎么算

`主问题`

MRR 一般看第一条正确结果，那如果一个问题有多个正确证据，你们怎么算？

`继续追问`

是不是很容易人为挑一个对你们有利的定义？

`参考回答`

更稳的做法是看“第一条命中的正确证据”排在第几位。也就是说，评测里先定义一组可接受的正确证据，只要第一个命中的正确证据出现在第 k 位，这条样本的 reciprocal rank 就是 `1/k`。这样不会强行要求命中特定唯一文档，也更贴近真实场景。你说得对，这个指标确实容易被口径影响，所以我面试里不会只报一个 MRR 数字，还会强调评测集构造方式和 badcase 分析，避免给人一种只是在调指标定义的感觉。

---

## 题 103：参数 EM 会不会太苛刻，为什么不用 F1

`主问题`

你们用参数 `EM`，是不是太严格了？少一个字段就全错？

`继续追问`

如果模型只错一个非关键字段，也算失败吗？

`参考回答`

对 Function Calling 来说，`EM` 严格一点是合理的，因为很多参数不是“差不多就行”，而是必须准确。比如项目、版本、任务、模块这些字段错一个，实际调用结果就可能完全变掉。所以我会把 `EM` 作为核心指标，反映结构化参数是否完全匹配。同时在内部分析时也可以补一个字段级准确率或 slot accuracy，用来判断到底是漏了哪个字段、错得有多严重。也就是说，`EM` 负责做上线口径，字段级指标负责做问题定位。

## 题 104：多工具场景里，工具选择准确率怎么定义

`主问题`

一个请求可能要调多个工具，那你说的工具选择准确率具体怎么算？

`继续追问`

是只看第一个工具，还是看整条链路？

`参考回答`

这要分层看。单步工具选择准确率，通常是看某一步该不该调工具、该调哪个工具。多步场景下，我会把它拆成单步准确率和链路级准确率。单步准确率看每一步是否选对工具，链路级准确率则看最终工具序列是否满足任务目标。因为在多步链路里，第一个工具选对不代表整条链就对了，所以我不会只报一个模糊的“工具准确率”，而会区分单步和端到端。

---

## 题 105：如果模型一开始选错工具，后来自己修正了，算成功吗

`主问题`

模型先错一次，后来又改对了，这种 case 你算成功还是失败？

`继续追问`

如果最后结果是对的，为什么还要扣它分？

`参考回答`

这取决于你的评估目标。如果评的是最终任务完成度，那它可以算端到端成功；但如果评的是工具决策质量，我还是会把前面的错误决策记下来。因为在企业场景里，中间多走一步错链路不仅增加延迟，也可能带来额外风险，尤其是涉及写操作时更不能只看最后结果。所以更稳的做法是同时记录：端到端是否成功，过程中有没有无效调用、错误调用和重复调用。这样评估才完整。

---

## 题 106：同一个问题今天问和明天问答案不一致，怎么解释

`主问题`

Agent 这种系统经常有不稳定性，同一个问题多问几次结果可能不一样，你怎么处理？

`继续追问`

业务方会觉得这很不可靠。

`参考回答`

先要区分“不一致”来自哪里。如果是底层业务数据变了，那结果变化是合理的；如果底层数据没变，但输出差异很大，就要怀疑检索候选、Prompt、采样参数或工具顺序不稳定。工程上我会尽量压低不必要的随机性，比如固定温度、稳定检索链路、限制工具空间、加强结构化输出约束。对业务方来说，最重要的是核心事实和结论要稳定，措辞细微不同反而不是主要问题。

---

## 题 107：多轮对话里的 memory 会不会把后面的决策带偏

`主问题`

你们提到 Memory，那它会不会把过期上下文带到下一轮，反而误导模型？

`继续追问`

比如用户已经换了版本，但模型还沿用上一轮版本。

`参考回答`

这个风险很现实，所以 memory 不能简单等于“把所有历史都喂进去”。我更认可的是分层记忆，保留对当前任务真正有约束价值的上下文，比如项目、版本、模块、任务范围，但要允许新输入覆盖旧上下文。另外要把高时效信息优先交给实时工具确认，而不是只靠对话历史继承。换句话说，memory 的目标是减少重复确认，不是把旧状态无限延续下去。

---

## 题 108：并发场景下，不同用户或不同会话怎么隔离

`主问题`

如果同一个用户同时开多个会话，或者多个用户并发使用，怎么避免上下文串掉？

`继续追问`

这不是很容易把 memory 搞乱吗？

`参考回答`

所以会话上下文一定要有明确的隔离键，至少区分用户、会话、项目和当前任务范围。工程上不应该把 memory 做成一个全局大对象，而应该是按 session 或 task 维度管理。对于工具调用日志、状态缓存、待执行任务，也都要带唯一上下文标识。模型本身不负责隔离，隔离是后端系统设计要解决的问题。

---

## 题 109：如果工具返回空结果，但模型还是硬答了，怎么控制

`主问题`

有时候接口没查到东西，模型还是会一本正经往下说，这种怎么拦？

`继续追问`

你不能只说“Prompt 约束一下”。

`参考回答`

对，这种问题不能只靠 Prompt。更稳的方式是把工具返回结果结构化，比如明确区分“调用成功但无数据”“调用失败”“权限不足”“参数错误”这几种状态，然后让 Agent 根据状态走不同分支。尤其当结果为空时，系统应该优先返回“当前没有查到相关数据，需要补充条件或检查上下文”，而不是让模型自由发挥。这里真正起作用的是工具协议设计和分支控制，不只是提示词。

---

## 题 110：OCR/ASR 本身可能错，为什么你还敢把截图和视频接进知识库

`主问题`

截图、视频、音频转文本本来就不稳定，你接进来不会引入更多噪声吗？

`继续追问`

那为什么不干脆不用这些数据？

`参考回答`

关键不是“接不接”，而是“怎么接”。我不会把 OCR/ASR 结果当成和结构化业务数据同等可信的事实源，而是把它当成补充证据。接入时也应该带来源类型、置信度、时间点和原附件引用，必要时降低其排序权重。因为这些多模态材料虽然噪声更高，但很多时候恰恰能补足流程里缺失的现场信息，所以不是不用，而是要带着质量分层和来源意识去用。

---

## 题 111：为什么你们没做知识图谱，而是做 metadata + 检索

`主问题`

测试平台里实体关系挺强的，为什么不直接做知识图谱？

`继续追问`

是不是你们方案太“轻”了？

`参考回答`

不是不能做，而是要看成本和收益。知识图谱更适合实体关系稳定、关系查询价值很高的场景；但你这个项目里有大量非结构化文档、评审意见、历史经验和多模态材料，单靠图谱不够覆盖。相对来说，先把 metadata 建好、检索链路做稳、工具抽象清楚，能更快形成可用系统。后面如果发现某些实体关系查询特别高频，再局部引入图谱会更合理。我的取舍不是否定图谱，而是优先做更高性价比的基础能力。

---

## 题 112：为什么是 SFT 再到 GRPO，而不是先做偏好优化

`主问题`

你们为什么先做 SFT，再考虑 GRPO？为什么不是直接上 DPO 或别的偏好优化？

`继续追问`

是不是因为你们没有更系统的训练能力？

`参考回答`

核心不是能力够不够，而是问题分层。对这个项目来说，第一步要先解决模型会不会稳定选工具、抽参数、输出结构化结果，这更像 supervised alignment，所以先做 SFT 更自然。等基础动作能力比较稳了，再考虑多步链路里的相对优劣，比如顺序、冗余调用和任务完成度，这时再引入 GRPO 之类的相对优化才有意义。如果基础动作都没站稳，直接做偏好优化通常收益有限。

---

## 题 113：训练样本来自历史人工操作，那不是会把人的坏习惯也学进去

`主问题`

你们拿历史操作链路做数据，怎么保证不会把低质量人工习惯学进去？

`继续追问`

比如有人本来就会乱填参数、乱走流程。

`参考回答`

这点必须承认，历史操作数据不是天然高质量。比较稳的做法是先筛样本，而不是全量照抄。优先选结果稳定、流程规范、业务认可度高的操作链路作为正样本，再把明显低质量链路转成负样本或 badcase。也就是说，历史数据是起点，不是金标准。真正的训练集应该是“历史行为 + 人工筛选 + 规则校验”共同构成的。

---

## 题 114：badcase 回流是按什么节奏做的

`主问题`

badcase 回流你说了很多次，那它到底是日常机制，还是偶尔人工看看？

`继续追问`

如果只是偶尔看一下，其实意义不大。

`参考回答`

对，badcase 回流如果没有节奏，就很容易流于口号。更稳的机制是按固定周期做，比如日常收集、每周归因、双周或月度合并进规则和训练集。高风险问题优先快速修，比如越权、错误写入、严重误导；中低风险问题则可以按批次进入下一轮优化。回流机制的关键不是“记录了多少问题”，而是问题有没有进入后续规则、数据和评测版本里。

---

## 题 115：你们怎么做 A/B test，而不是只靠离线指标拍脑袋

`主问题`

如果线上要比较新旧方案，你们怎么做 A/B？

`继续追问`

A/B 的胜负标准看什么？

`参考回答`

我会优先选真实高频场景做受控对比，比如测试用例初稿生成、风险摘要、信息汇总这种使用频率高又相对可比的任务。A/B 不只看单一模型指标，而要看采纳率、人工改写率、任务耗时、回退率和异常调用类型。如果新方案离线更好，但线上采纳率没变甚至更差，那就说明它未必更适合真实场景。对 Agent 项目来说，A/B 最终比的是业务可用性，不只是模型分数。

---

## 题 116：灰度放量时，你决定继续放量的门槛是什么

`主问题`

你怎么决定一个 Agent 能从小流量继续放大？

`继续追问`

总不能只靠“感觉还不错”。

`参考回答`

我会同时看三类信号。第一是基础稳定性，错误率、超时、异常调用不能失控。第二是链路质量，空召回率、参数缺失率、重复调用率不能明显恶化。第三是业务效果，采纳率、人工改写率、回退率要达到预设门槛。只有这三层都达标，才值得继续放量。灰度的价值就是把“感觉不错”变成可量化的准入条件。

---

## 题 117：如果线上真出事故了，你第一反应做什么

`主问题`

假设系统误生成了错误结论或者触发了错误写入，你第一步做什么？

`继续追问`

不要只说“看日志”。

`参考回答`

第一步是止血，也就是先关闭高风险链路，必要时切回只读辅助模式。第二步是定界，确认影响范围，是某个 Prompt、某个工具、某个版本还是整条链路的问题。第三步才是基于调用日志、输入输出、工具返回和权限记录做回放定位。出了事故先控制风险，再谈定位根因，这是复杂系统最基本的处理顺序。

---

## 题 118：如果面试官怀疑这些指标不是严格实验，而是估算出来的，怎么答

`主问题`

面试官可能会质疑你简历上的指标只是“项目汇报口径”，不够严格，你怎么回应？

`继续追问`

如果他继续追问实验设计细节呢？

`参考回答`

我会先承认指标必须有口径前提，然后把评测设计讲清楚。比如 Recall 和 MRR 来自离线标注的 query-evidence 评测集，工具选择准确率和参数 EM 来自独立测试集，业务效率提升来自人工流程和 Agent 辅助流程的耗时对比。这样先把指标类型和测量方式说开，比直接背数字更有说服力。如果面试官继续追，我就继续讲样本来源、切分方式、训练评测隔离和 badcase 分析。越能把实验口径讲清楚，越不像“拍脑袋估算”。

---

## 题 119：如果用户说“你这个结果看起来对，但我还是不敢用”，你怎么改

`主问题`

业务方不一定说你错，他可能只是说“不敢用”，这种怎么处理？

`继续追问`

这不是准确率问题吧？

`参考回答`

对，这更多是信任问题，不只是准确率问题。通常要补三件事。第一是引用和依据，让用户知道结论来自哪些规划文档、缺陷趋势或工具查询结果。第二是结构化输出，把结论、证据、风险和建议动作拆开，不要混成一段大话。第三是人工确认点，让系统在关键场景里停在“辅助决策”而不是直接替人做决定。很多时候用户不是怀疑你完全错，而是担心错了之后没人能解释。

---

## 题 120：如果让你把这套方法迁到另一个业务线，你第一步做什么

`主问题`

如果以后不是做测试平台，而是迁到别的业务线，你第一步会做什么？

`继续追问`

是不是把 RAG 和工具接过去就行？

`参考回答`

第一步不是搬技术，而是重新定义任务边界。先找清楚那个业务线里什么动作最重复、什么知识最分散、什么接口最值得 Agent 化，然后再决定知识链路、工具集合和评测口径。因为可迁移的是方法，不是原封不动的配置。RAG、Tool Schema、Agent Orchestrator 这些都是通用壳子，但真正决定效果的是业务目标和评测边界有没有重新定义对。

## 题 81：如果只能先做一个 MVP，你会先做哪部分

`模拟问题`

这个项目能做的事情很多，如果只能先做一个最小可用版本，你会怎么砍范围？

`参考回答`

我会优先做测试用例初稿生成和知识辅助检索，不会一开始就上高风险自动写入。因为这两个场景价值清晰、频率高、验证成本低，而且业务方更容易接受。MVP 的目标不是证明系统什么都能做，而是先选一个高频、低风险、容易量化收益的切口，把采纳率和使用习惯跑出来。等这个环节跑通，再逐步扩到执行编排和质量诊断，风险会小很多。

## 题 82：如果领导问这个项目的 ROI，你怎么回答

`模拟问题`

你怎么证明这个 Agent 项目值得继续投入？

`参考回答`

我会把 ROI 拆成直接收益和间接收益。直接收益是人效，比如测试用例初稿生成耗时下降、资料收集时间下降、风险汇总时间下降。间接收益是流程标准化和知识沉淀，比如减少经验依赖、降低新同学上手成本、让历史项目知识可检索可复用。对领导来说，最有说服力的不是模型指标，而是它能不能稳定减少人工重复劳动，能不能把流程里的隐性成本显性化。

## 题 83：你怎么决定第一批要接哪些工具，而不是一口气全接

`模拟问题`

平台里接口很多，你怎么决定 Agent 第一阶段要接哪几个？

`参考回答`

我会按“调用频率、价值密度、风险等级”三个维度筛。优先接的是高频、强价值、低副作用的读工具，比如任务查询、缺陷统计、版本信息、历史用例检索、测试报告摘要。这类工具最容易形成业务闭环，也最容易积累 badcase。高风险写工具不会一开始全接，因为它们对权限、幂等和回滚要求更高。我的取舍原则是先把高频查询和辅助生成做稳，再逐步开放更深的执行能力。

## 题 84：评测集和 badcase 标注通常很耗人力，你怎么推动这件事

`模拟问题`

业务团队通常不愿意花时间帮你做评测和标注，你怎么推进？

`参考回答`

我不会一上来就让业务方做大规模纯标注，这很难推动。更现实的方式是把标注嵌进现有流程里，比如把“采纳、修改、拒绝、补充说明”这些行为变成轻量反馈，再从高频 badcase 里抽样做精标。这样既不会额外制造太多负担，也能逐步积累真实评测数据。做 Agent 项目时，最好的标注体系通常不是独立造一个流程，而是顺着原有业务动作把数据收回来。

## 题 85：如果测试同学和后端同学对项目目标理解不一致，你怎么协调

`模拟问题`

测试团队更关心结果好不好用，后端团队更关心接口成本，你怎么对齐？

`参考回答`

这种项目很容易两边各说各话，所以我会先把目标拆成可共识的几项。对测试团队，我会强调是否省时间、结果是否可用、能不能减少重复劳动。对后端团队，我会强调工具边界、权限治理、接口复用和服务稳定性。真正的对齐不是让所有人关心同一件事，而是把项目拆成他们各自认可的价值点，再用一套共同指标把这些点串起来。

## 题 86：如果产品经理说“这个功能看起来很酷，但不是刚需”，你怎么回应

`模拟问题`

面对对 AI 项目持保留态度的产品经理，你怎么讲？

`参考回答`

我不会先讲模型有多先进，而是先讲它替代的是哪一段重复劳动。比如测试用例初稿、风险摘要、资料汇总，本来就要有人花时间做，只是现在把这段工作前移成了机器辅助。这样项目就不再是“多了一个酷功能”，而是“把原来就存在的成本降下来”。如果一个 AI 功能找不到明确被替代的动作，它通常很难成为刚需。

---

## 题 87：如果项目资源有限，你怎么排期

`模拟问题`

一个人或小团队做这种项目，怎么安排优先级和节奏？

`参考回答`

我一般会按“三段式”排期。第一段先打底，包括知识链路、工具抽象、评测口径和日志观测，不把地基打稳，后面很难持续迭代。第二段做高频低风险场景上线，优先拿到真实反馈。第三段再针对 badcase 做精细优化，包括 rerank、SFT、参数抽取和链路反思。排期的核心不是把所有能力平铺推进，而是每一阶段都能形成一个可验证的小闭环。

---

## 题 88：如果领导只让你每周汇报一个指标，你汇报什么

`模拟问题`

项目指标很多，如果只能保留一个周指标，你选哪个？

`参考回答`

我会优先报“有效采纳率”或者“人工改写后采纳率”，因为这是最接近真实业务价值的指标。单看 Recall、MRR 或工具准确率只能说明局部能力，但领导更关心这个系统有没有在真实流程里被持续使用。如果每周只能看一个指标，我宁愿看业务层面的采纳和回退，也不只看模型层面的离线分数。

---

## 题 89：如果你只有一周准备给老板做演示，你怎么取舍

`模拟问题`

离汇报只剩一周，你怎么保证演示既稳又有说服力？

`参考回答`

这种时候我不会追求全，而会追求稳和可解释。优先选一到两个最有代表性的高频场景，准备固定输入、固定知识材料、固定工具链路，把演示链条走通。同时一定要准备指标、badcase 前后对比和回滚方案，不只是做一个“看起来很聪明”的 demo。对老板来说，能稳定复现、能解释为什么有效，比临时堆很多花活更有价值。

---

## 题 90：如果项目中途发现最初假设错了，你怎么调整

`模拟问题`

比如你原来以为主要瓶颈是模型能力，后来发现其实是数据质量问题，你怎么办？

`参考回答`

我会尽快把方向纠正回来，而不是为了保持一致性继续沿错路投入。Agent 项目很容易一开始把问题都归到模型上，但真实场景里，很多核心问题其实在数据、工具边界和流程设计。发现假设错了之后，应该及时重排优先级，把资源转回最影响上限的环节。工程上，及时承认方向需要调整，比继续优化错误目标更专业。

---

## 题 91：如果别的团队想复用你们这套能力，你觉得哪些是通用的

`模拟问题`

这个项目是不是只能用于测试平台，还是能抽成更通用的能力？

`参考回答`

我会把它拆成“业务专属层”和“通用能力层”。像测试设计、缺陷趋势、TR 报告这些是测试场景专属的，但 DataAgent、混合检索、Tool Schema、Agent Orchestrator、评测闭环、badcase 回流这些其实是通用能力。也就是说，这个项目不只是做了一个测试 Agent，而是验证了一套“企业私有知识 + 工具调用 + 业务工作流”的工程方法论，这部分是可以迁移的。

---

## 题 92：如果面试官问“为什么不直接买外部现成产品”，你怎么答

`模拟问题`

市面上也有很多 Agent 平台或测试助手，你们为什么要自己做？

`参考回答`

我会从三点回答。第一，核心数据在企业内部，外部产品很难天然理解项目、版本、模块、任务这些上下文。第二，平台里已经有大量现成 API 和业务流程，真正有价值的是把这些资产 Agent 化，而不是买一个脱离现有系统的外部工具。第三，测试场景对权限、审计和流程嵌入要求很高，很多外部产品只能做问答，未必能安全进入业务闭环。所以不是不能买，而是核心价值恰恰在于和内部系统深度结合。

---

## 题 93：如果法务或安全同学介入，这个项目最容易被问什么

`模拟问题`

从合规和安全角度，他们最可能卡你哪些点？

`参考回答`

最容易被问的是三类。第一，数据边界，哪些文档、附件、缺陷信息可以被检索和调用。第二，权限控制，模型是不是可能越权拿到不该看的项目数据。第三，输出和调用审计，尤其是写接口动作是否能回溯。我的应对思路是把权限放在服务端兜底，把检索和工具都纳入统一鉴权，同时保留调用链和结果引用，确保系统不是一个不可追踪的黑盒。

---

## 题 94：如果 GPU 或模型资源紧张，你怎么保证项目还能推进

`模拟问题`

资源不够的时候，你怎么避免整个项目卡住？

`参考回答`

我会优先保住高价值链路，而不是默认所有环节都要最强模型。比如检索、metadata 过滤、工具适配、规则校验这些很多都不依赖大 GPU；复杂任务再走重模型，简单任务走轻模型或模板链路。项目推进的关键是先把工程闭环跑起来，而不是等最理想的模型资源到位。很多时候，资源紧张更能逼出更合理的分层设计。

---

## 题 95：如果业务方对“质量诊断”结果意见很多，你怎么收敛

`模拟问题`

质量诊断这种事情本来就带主观判断，结果风格不统一怎么办？

`参考回答`

这种问题不能只靠模型继续学，而要先把输出框架定下来。比如固定按风险等级、证据来源、影响范围、建议动作这几个部分输出，让“结构先统一”，再去优化内容细节。另外可以让测试负责人先定义一版可接受模板，把主观分歧收敛成几个明确维度。否则如果没有统一结构，后面无论是 Prompt、SFT 还是 GRPO，都会因为目标不清而很难优化。

---

## 题 96：上线前你会做哪些验证，而不是直接灰度

`模拟问题`

一个复杂 Agent 项目在灰度前，最少要做哪些验证？

`参考回答`

至少要覆盖四类。第一是离线评测，确保检索、工具、参数抽取和端到端任务在固定评测集上可回归。第二是链路压测，看重排、工具调用和多步编排在真实并发下是否稳定。第三是权限和异常测试，确认越权、超时、空结果、重复写入这些场景能被拦住。第四是回滚演练，确保一旦效果抖动能迅速退回安全模式。灰度不是拿来替代基础验证的，而是在基础验证之后观察真实使用分布。

---

## 题 97：如果这个项目没有达到预期，你怎么复盘

`模拟问题`

如果最后效果一般，你会怎么复盘，而不是泛泛地说“继续优化”？

`参考回答`

我会按“目标、路径、证据、决策”四层复盘。先看目标是不是定义清楚了，比如到底要提升的是人效、质量还是流程标准化。再看路径是否合理，是不是过早投入模型优化，而忽略了数据和工具。然后看证据，哪些指标支持结论，哪些只是感觉。最后看决策，哪些投入是必要的，哪些是当时判断失误。这样复盘出来的是下一轮可以执行的结论，而不是一句空泛的“模型还不够强”。

---

## 题 98：如果让你给这个项目定三个里程碑，你怎么定

`模拟问题`

项目从 0 到 1 再到 1 到 N，你会怎么拆里程碑？

`参考回答`

第一阶段是 0 到 1，目标是打通知识链路、工具抽象、评测体系，做出可运行 MVP。第二阶段是 1 到可用，目标是选一两个高频场景灰度上线，把采纳率、改写率、badcase 类型跑出来。第三阶段是可用到可扩展，目标是做权限治理、分层回滚、观测看板和更多 Agent 协作场景。这样拆的好处是每一阶段都有明确验收标准，而不是只有一个模糊的大目标。

---

## 题 99：你在这个项目里最能体现“工程判断力”的一个决定是什么

`模拟问题`

如果只能举一个例子证明你不是只会堆技术，你会举什么？

`参考回答`

我会举“没有把问题一股脑推给模型”这件事。这个项目里很多表面问题都可以解释成模型不够强，但我更倾向于先拆数据、检索、工具、流程和权限边界，把系统做成可诊断、可优化的结构。比如先把 Tool Schema、评测闭环和 badcase 回流做起来，再决定哪里值得上 SFT 或 GRPO。这种取舍本质上体现的不是某个单点技术，而是工程判断力。

---

## 题 100：如果让你用一句话总结这个项目里你的价值，你怎么说

`模拟问题`

面试快结束了，面试官让你用一句话总结自己在这个项目里的价值，你怎么答？

`参考回答`

我会说，我的价值不是单点做了某个模型优化，而是把企业测试平台里的私有知识、后端工具和 Agent 工作流真正接成了一个可评测、可灰度、可上线的工程闭环。

## 说明

这一批继续围绕你的主项目：

`基于 RAG + Function Calling 的游戏智能测试 Agent 平台`

前面三批已经覆盖了项目介绍、架构拆层、高压追问。这一批重点补下面四类问题：

## 题 61：知识库和线上业务数据之间的数据时效差，你怎么处理

`模拟问题`

知识库通常不是实时的，如果测试平台里的状态已经变了，Agent 还在引用旧知识怎么办？

`参考回答`

我会把“事实查询”和“背景知识”分开处理。像任务状态、缺陷数量、执行进度这种强时效信息，我优先走实时工具查询，不直接信知识库。知识库更适合承载规划、历史经验、评审意见、风险案例这类相对稳定的信息。工程上还要做两件事，一是给知识对象带时间、版本和来源字段，二是在输出阶段把“实时工具结果”和“历史检索证据”分开展示。这样能降低旧知识误导当前结论的风险。

---

## 题 62：chunk 大小怎么选，你怎么证明切分策略合理

`模拟问题`

RAG 里 chunk 大小经常影响效果，你们是怎么选的？

`参考回答`

我不会把 chunk 大小当成固定魔法数字，而是按文档类型来定。像版本规划、评审意见这种结构比较清晰的文档，更适合按章节或语义段切；历史测试用例和缺陷分析则更适合按条目或步骤切。核心不是切多大，而是切完之后每个 chunk 还能不能保持完整语义、能不能被 metadata 正确约束、能不能在 rerank 阶段保持可解释性。证明方式也不是拍脑袋，而是做离线评测，看不同切分策略下 Recall@10、MRR、长尾 query 命中率和 badcase 类型怎么变化。

---

## 题 63：如果 Query Rewrite 改写错了，反而把检索带偏了，怎么办

`模拟问题`

Query Rewrite 有时会“自作聪明”，你怎么控制这个风险？

`参考回答`

这个风险一定存在，所以 Query Rewrite 不能是黑盒单点决策。我更认可的做法是保留原始 query，同时生成 rewrite 版本，然后在召回和排序阶段做融合，而不是完全覆盖原问题。这样即使改写有偏差，也不至于把原始意图彻底丢掉。另外可以给 rewrite 加边界，例如只补项目、版本、模块、时间这些结构性约束，不随意扩写业务语义。真正出问题时，还能通过日志回放看到“原 query、改写后 query、最终命中文档”三者之间的关系，方便定位。

---

## 题 64：如果 metadata 过滤太强，把正确结果过滤掉了，你怎么发现

`模拟问题`

metadata filter 虽然能减少噪声，但也可能召回过窄，这怎么排查？

`参考回答`

我会同时看两类信号。第一类是离线评测里出现“明明库里有正确证据，但最终候选为空或明显变少”的样本，这通常说明过滤条件过严。第二类是线上 badcase，用户提问很明确，但系统总说“没有找到相关信息”，这也要怀疑过滤而不是只怀疑向量检索。排查时要把 Query Rewrite 生成的约束条件、metadata 实际过滤字段和值都打出来。很多时候问题不是检索模型弱，而是版本、模块、时间窗口这些过滤条件配错了。

---

## 题 65：为什么要做 rerank，不怕再加一层延迟吗

`模拟问题`

重排能提效果，但也增加延迟，你怎么证明值得做？

`参考回答`

值不值得，关键看它解决的是不是高价值错误。企业场景里的问题通常不是“一个相关结果都没有”，而是“候选里有几个都沾边，但顺序错了”。如果不做 rerank，模型很容易吃到一个表面相关但上下文不对的片段，后面整条链路都会偏。对这种问题，Cross-Encoder 重排的收益通常比单纯多召回几个候选更大。当然我不会让所有请求都走最重的 rerank，可以按场景分层，简单查询走轻量排序，复杂设计和风险诊断再走完整链路。

---

## 题 66：为什么不是纯 Elasticsearch，而是混合检索

`模拟问题`

企业里很多知识检索本来就能用 Elasticsearch，你为什么还要上向量检索？

`参考回答`

因为你的查询同时包含两种需求。第一种是精确匹配，比如版本号、任务名、模块名、缺陷类型，这部分 Elasticsearch 很强。第二种是语义理解，比如“类似历史版本里容易漏掉的风险点”“结合评审意见给出测试建议”，这种只靠关键词很难稳定命中。纯 Elasticsearch 会漏掉语义相关，纯向量又容易在精确实体上漂移，所以混合检索是更务实的方案。它不是为了追新技术，而是为了把精确匹配和语义匹配放在同一条链路里。

---

## 题 67：如果线上遇到长尾问题，工具和知识都没有覆盖，Agent 应该怎么退化

`模拟问题`

面对完全没见过的长尾问题，系统最稳的处理方式是什么？

`参考回答`

最稳的做法不是硬答，而是优雅退化。具体就是三步。第一，明确告诉用户当前证据不足，哪些上下文缺失。第二，如果能补充，就引导用户补项目、版本、模块、目标范围。第三，如果已经超出系统能力边界，就退回到只做信息汇总或建议，而不是继续走高风险工具链路。企业场景里，知道什么时候该停，比勉强给一个完整答案更重要。

---

## 题 68：多工具链路里某一步失败了，你怎么做重试和补偿

`模拟问题`

如果一个多步任务执行到中间失败了，系统怎么处理？

`参考回答`

首先要区分失败类型。查询类失败和写操作失败处理方式不一样。查询失败一般可以安全重试，但写操作必须考虑幂等和补偿。我的思路是，把多步链路拆成可观测的状态节点，每一步记录输入、输出和状态。对可重试的调用做有限次重试，对高风险写操作则优先设计幂等接口、请求唯一键和回滚策略。这样即使中间失败，也能知道失败点在哪、是否已经产生副作用，而不是整条链路黑盒化。

---

## 题 69：Agent 触发写接口时，怎么避免重复写入

`模拟问题`

模型有时会重复调用工具，创建任务、更新状态这类接口怎么防重复？

`参考回答`

这个问题不能只靠模型克制，必须靠后端兜底。最核心的是幂等设计，比如给写请求加唯一业务键、去重 token 或请求签名，同一操作重复提交时服务端直接识别为同一次动作。另外在 Agent 层也要限制重复规划，比如 Reflect 阶段检查当前动作是否已经执行成功。我的原则是：模型可以犯错，但后端不能因为模型犯错就把业务数据写坏。

---

## 题 70：你们上线后最该监控哪些指标

`模拟问题`

如果让你给这个系统做监控面板，你最先放哪些指标？

`参考回答`

我会分四层。第一层是基础稳定性，比如请求量、接口超时、错误率、平均耗时。第二层是检索层，比如空召回率、候选数量分布、重排耗时、引用命中率。第三层是工具层，比如工具选择准确率、参数缺失率、写接口拦截率、重复调用率。第四层是业务层，比如采纳率、人工改写率、回退率、测试用例初稿生成耗时。这样监控板不是只看系统活没活着，而是能看到它到底在哪一层开始变差。

---

## 题 71：Prompt 经常改，怎么做版本管理和效果对比

`模拟问题`

Prompt 一直在迭代，你怎么避免改一版坏一版？

`参考回答`

我会把 Prompt 当成配置资产管理，而不是当临时字符串改。至少要有版本号、变更说明、适用场景和离线评测结果。每次改动都应该在固定评测集上回归，看它影响的是检索、工具、还是端到端指标。如果是线上灰度，还要看采纳率和 badcase 分布有没有恶化。没有版本管理的 Prompt 优化，很快就会变成“感觉这版更好”，但实际上没人能证明。

---

## 题 72：如果成本压力很大，你怎么降 token 和算力开销

`模拟问题`

RAG、重排、反思、多工具调用都要钱，你怎么做成本控制？

`参考回答`

我会先找浪费最大的环节，而不是一上来就降模型。通常优先看三件事：有没有无效检索、有没有冗余工具调用、有没有不必要的长上下文。能通过 metadata 缩小范围的，就不要让模型读全库；能通过轻量规则做预筛的，就不要都交给大模型；简单任务能走轻模型的，就不要默认最重配置。成本控制的核心不是“全面降配”，而是让不同价值的请求走不同强度的链路。

---

## 题 73：你们怎么收集真实用户反馈，而不是只看日志

`模拟问题`

如果用户觉得结果不好，系统怎么知道是哪里不好？

`参考回答`

只看日志不够，因为日志只能看到系统行为，不能直接知道用户为什么不满意。比较实用的方式是把反馈收集嵌进业务流程，比如让用户对生成结果做采纳、修改、拒绝或补充说明，并把这些动作和当次链路日志关联起来。这样后面做 badcase 归因时，就不仅知道“结果错了”，还能知道是事实错误、结构不对、太啰嗦、还是不符合测试团队的使用习惯。

---

## 题 74：如果灰度上线后效果不稳定，你怎么回滚

`模拟问题`

Agent 项目灰度期间最怕效果忽高忽低，你怎么设计回滚机制？

`参考回答`

回滚不能只理解成“把模型版本切回去”，而是要支持分层回滚。比如可以单独关闭 rewrite、关闭 rerank、切回旧 Prompt、降级到只读辅助模式，或者直接退回到原有人工流程。这样出问题时不用整个系统一起下线，能更快定位到底是哪一层变坏了。对这种复杂链路系统来说，分层开关和分层回滚比单一总开关更有价值。

---

## 题 75：如果测试团队不买账，你怎么推动落地

`模拟问题`

技术上做出来了，但测试同学觉得麻烦、不敢用，这种项目怎么推进？

`参考回答`

我不会一开始就要求他们完全改变工作方式，而是先切低风险、高收益的入口。比如先做测试用例初稿、风险摘要、信息汇总这类辅助能力，而不是直接替他们做最终决策。同时要尽量把结果嵌回现有平台，让他们不用跳到另一个系统里工作。推动落地的关键不是证明 AI 很先进，而是证明它确实少做了很多重复劳动，而且不会把原来的流程搞乱。

---

## 题 76：如果面试官问“这里面到底哪些是你做的”，你怎么答得既真实又不显弱

`模拟问题`

这个项目内容很多，怎么讲自己的贡献边界最稳？

`参考回答`

我的建议是按“我主导 / 我重点参与 / 团队协作”三层说清楚。比如可以说，我主导的是后端侧 Agent 化落地，包括 DataAgent 到 RAG 的知识链路、Tool Schema 抽象、Agent 工作流接入和评测闭环；重点参与的是 Function Calling 的数据构造、离线评测和 badcase 回流；像更底层的模型训练框架、基础设施资源调度如果不是你主导，就明确说是团队协作。这样既真实，也不会显得你什么都只沾一点。

---

## 题 77：如果某个局部指标提升了，但端到端任务成功率下降，你先看什么

`模拟问题`

比如 Recall 提升了，但实际任务成功率反而降了，这种情况怎么解释？

`参考回答`

这很常见，说明局部最优不等于全链路最优。比如召回更多了，但噪声也更多，rerank 或模型没把真正相关的信息排到前面，最终反而干扰了工具调用。又或者为了提高召回，metadata 约束放松了，结果混进了跨版本内容。我的排查顺序一般是：先看候选质量，再看排序，再看模型吃进去的上下文长度和工具决策变化。不能因为一个局部指标涨了，就默认系统整体更好。

---

## 题 78：badcase 很多时，你怎么排优先级

`模拟问题`

如果线上 badcase 一抓一大把，你先修哪些？

`参考回答`

我不会按“看起来最难受”来排，而是按业务风险和出现频次一起排。第一优先级一定是会导致错误写入、误判风险或严重误导业务的 case。第二类是高频发生、明显影响采纳率的问题，比如参数经常漏、引用经常缺。第三类才是长尾但影响相对可控的问题。很多团队容易先修最有意思的问题，但工程上更应该先修最伤系统信任的问题。

---

## 题 79：如果资源有限，只能先做一部分能力上线，你会砍掉什么

`模拟问题`

这个项目功能很多，如果必须砍范围，你优先砍哪一部分？

`参考回答`

我会优先砍高风险、低复用、强自动写入的能力，先保留只读辅助和高频场景。比如先做知识问答、测试用例初稿、风险摘要，不急着上线高风险写接口和过深的自动编排。这样能先把用户信任、数据闭环和观测体系建立起来，后面再逐步加自动执行能力。做复杂系统时，收缩范围往往比一次把所有能力堆上去更专业。

---

## 题 80：如果面试官说“这不就是系统集成吗，没有技术含量”，你怎么回应

`模拟问题`

很多人会低估 Agent 工程，说这只是把现成模型和接口拼起来，你怎么反驳？

`参考回答`

我不会去争概念，而是直接把技术难点拆开讲。这个项目的难点从来不是“能不能把模型调通”，而是怎么把企业私有知识做成可检索体系，怎么把平台 API 抽象成稳定 Tool Schema，怎么让多工具链路在参数、权限、日志、回滚和 badcase 回流上形成闭环。表面看像集成，真正难的是把每一层都做得可控、可评估、可上线。很多 demo 都能拼出来，但能不能进入真实测试流程，差距就在这些工程细节里。

## 题 41：如果检索结果和工具返回结果冲突，你信谁

`模拟问题`

如果 RAG 检索出来的知识和平台实时接口返回的数据不一致，你最终按哪个来？

`参考回答`

我会优先信实时工具结果，把检索结果当作背景知识和解释依据，而不是最终事实源。因为知识库天然有同步延迟，尤其是任务状态、缺陷数量、执行进度这类强时效数据，更应该以平台实时 API 为准。比较稳的做法是把两者职责拆开：RAG 负责提供上下文、历史经验和解释材料，工具负责提供最终状态和可执行结果。这样即使知识库有轻微滞后，也不会把系统结论建立在过期事实上。

## 题 42：如果离线指标很好，但线上采纳率不高，你怎么解释

`模拟问题`

你简历里离线指标提了很多，如果线上大家不用，说明什么？

`参考回答`

这通常说明离线指标只覆盖了“模型能力”，但没有覆盖“真实使用成本”。常见原因有三类。第一类是任务定义没对齐，离线测的是检索或工具调用，但线上用户关心的是能不能真正减少工作量。第二类是交互链路太重，结果虽然对，但太慢、太啰嗦、太难接入现有流程。第三类是结果可解释性不够，业务方不敢直接采用。所以我会把离线指标和线上指标拆开看：离线看 Recall、MRR、工具准确率、参数 EM，线上看采纳率、人工改写率、回退率、耗时下降和 badcase 类型分布。只有两层指标都成立，项目价值才算闭环。

---

## 题 43：为什么不直接做一个规则工作流，而一定要上 Agent

`模拟问题`

测试平台本来就能做流程编排，为什么不继续用规则引擎，非要上 Agent？

`参考回答`

因为这个场景的问题不只是“按固定顺序调接口”，而是要先理解自然语言需求，再决定查什么知识、调什么工具、补什么上下文。规则工作流适合边界清晰、输入结构稳定的任务，但测试设计、风险评估、质量诊断这类场景输入非常开放，很多信息藏在规划文档、历史缺陷和多轮对话里。Agent 的价值就在于能在不确定输入下做动态决策。当然我不会把所有事情都交给 Agent。凡是规则明确、确定性强的步骤，还是应该固化成工具或工作流节点，Agent 只负责高变动、难写死规则的那部分决策。

## 题 44：为什么不直接全靠微调，不做 RAG

`模拟问题`

既然你们已经做 SFT 和 GRPO 了，为什么不继续把知识也学进模型里？

`参考回答`

因为你这个场景里的核心知识不是稳定常识，而是企业内动态变化的数据。像版本规划、任务状态、缺陷趋势、TR 报告、评审意见这些内容更新很快，而且很多都带明确时间和版本上下文，不适合靠模型参数记忆。把这些内容硬塞进模型，一方面更新成本高，另一方面很容易过期。更合理的做法是把“知识获取”交给 RAG，把“动作决策”交给 Function Calling，把“稳定调用能力”再通过 SFT 或 GRPO 优化。这样知识和能力是解耦的，后续迭代成本也更低。

## 题 45：你怎么定义这个 Agent 项目到底算成功

`模拟问题`

如果你是项目负责人，你会怎么定义这个项目做成了？

`参考回答`

我会把成功拆成三层。第一层是能力层，系统至少能稳定完成检索、工具选择、参数抽取和结果汇总。第二层是流程层，系统真的进入了测试设计、执行编排或质量诊断链路，而不是停留在问答 demo。第三层是业务层，能明确降低人工成本，比如测试用例初稿生成时间下降、人工收集信息时间下降、评审前准备时间下降，同时 badcase 可回溯、风险可审计。对我来说，只有“能被业务持续使用”才算成功，不是看几个 demo 演示得好不好。

## 题 46：离线评测怎么尽量贴近线上真实场景

`模拟问题`

很多离线评测都很理想化，你怎么让评测更接近线上？

`参考回答`

我会优先做三件事。第一，样本来源尽量贴近真实输入，不只用人工构造的标准问句，也要覆盖口语化、歧义化、多轮上下文和不完整输入。第二，切分方式尽量按时间或场景隔离，避免训练和评测里只是同模板改写。第三，评测粒度要分层，既看检索层和工具层指标，也看端到端任务成功率。这样评测才能更像真实线上，而不是只在一个理想模板集上“刷分”。

## 题 47：如果工具选择准确率上去了，但用户还是觉得不好用，问题可能出在哪

`模拟问题`

工具选对了，不代表系统好用。那剩下的问题通常是什么？

`参考回答`

最常见的是三类。第一类是前置检索质量不够，工具虽然选对了，但输入上下文本身就不对。第二类是参数不完整，工具名对了，但项目、版本、模块这些关键字段漏了，实际结果依旧不可靠。第三类是结果表达不符合业务预期，比如输出太长、没有引用、不能直接复用到测试流程里。所以我不会把工具选择准确率当成最终目标，它只是中间指标。真正要看的还是任务成功率、人工改写率和采纳率。

## 题 48：你觉得这个项目里最难的点是什么

`模拟问题`

如果只让你说一个最难点，你会说什么？

`参考回答`

我会说最难的是把“看起来会回答”的系统做成“真的能稳定执行”的系统。因为问答类 demo 最大的问题是只要输出像回事就行，但你的场景里很多动作是有后果的，比如生成测试建议、汇总风险、甚至触发写接口。真正难的不是让模型多聪明，而是让整条链路可控，包括知识质量、工具边界、参数完整性、权限约束、日志审计和 badcase 回流。这些事情单看都不是模型算法本身，但少了任何一层，系统都很难上线。

---

## 题 49：怎么避免 Agent 一本正经地胡说八道

`模拟问题`

你们这个系统怎么尽量抑制幻觉？

`参考回答`

我会从三个层面控制。第一是知识层，尽量要求回答建立在检索命中的内部材料之上，关键结论要有引用，不让模型纯凭先验编。第二是工具层，涉及状态查询和业务事实时优先走实时接口，不让模型自己猜。第三是输出层，信息不足时允许明确说“当前证据不足，需要补充上下文”，而不是强行给完整答案。对这种企业场景来说，少说错话比多说漂亮话更重要。

---

## 题 50：为什么不直接上长上下文模型，少做一层检索

`模拟问题`

既然现在长上下文模型很多，为什么还要做 RAG？

`参考回答`

长上下文解决的是“能装下更多文本”，但不等于“能稳定找到最相关的证据”。你的场景里数据来源杂、版本多、时效强，如果直接把大量文档堆进上下文，一方面成本高、延迟高，另一方面噪声也会明显增加。RAG 的价值不是单纯节省 token，而是先做筛选，把真正相关的上下文挑出来，再交给模型。长上下文可以作为补充，但一般不应该替代检索。

---

## 题 51：为什么不做端到端多模态，而只是把截图视频转成可检索文本

`模拟问题`

你们有截图和视频，为什么不直接做多模态大模型端到端理解？

`参考回答`

如果目标是研究更强的多模态理解，端到端当然有价值。但从工程落地角度看，我更关心的是稳定性、成本和可检索性。截图和视频在这个项目里主要承担的是补充证据和还原现场，不一定每次都值得走完整多模态推理链。先通过 OCR、ASR、关键帧抽取把内容转成可检索文本和结构化元信息，能够更稳定地进入知识链路，也更容易做引用、过滤和回溯。这是更务实的取舍。

---

## 题 52：不同项目、不同版本的数据怎么做权限隔离

`模拟问题`

如果是企业内部平台，权限和数据隔离怎么做？

`参考回答`

我会把权限控制放在工具层和检索层同时做，而不是只靠前端隐藏。检索层要基于用户身份、项目归属、版本范围做 metadata 过滤，避免不该看到的数据被召回。工具层则要按权限分读写级别，尤其是写接口，必须做服务端鉴权和参数校验。这样即使模型本身“想调”，真正能不能看到、能不能写，还是由后端权限系统决定，而不是由模型决定。

---

## 题 53：如果知识库里的不同文档彼此矛盾，Agent 怎么处理

`模拟问题`

规划文档、评审意见、历史周报经常互相打架，这时候系统怎么给结论？

`参考回答`

这种情况不能简单求一个“平均答案”。更稳的做法是保留来源和时间信息，让系统先识别冲突，再按优先级判断。一般当前版本正式规划、最新评审结论、实时平台状态会优先于旧周报或历史经验。如果冲突还无法消解，系统应该显式指出“当前资料存在冲突”，并列出冲突来源，而不是自己拍脑袋选一个。企业场景里，识别冲突本身就是价值。

---

## 题 54：平台 API 经常变，Tool Schema 怎么维护

`模拟问题`

如果后端接口字段老变化，Tool Schema 不会很快过时吗？

`参考回答`

会，所以 Tool Schema 不能只是一次性手写配置，而要当成平台契约来维护。比较稳的做法是把工具定义尽量贴近后端接口规范，并给参数做版本管理和兼容层。对高频变化接口，可以在 Agent 层只暴露稳定抽象，把底层字段变化收敛在服务端适配层里。这样模型学习的是稳定业务动作，而不是直接绑定某个随时会改的内部字段名。

---

## 题 55：你怎么证明 Recall@10 和 MRR 不是“刷出来”的

`模拟问题`

你这些检索指标提升很多，面试官可能会怀疑是不是评测集太容易，你怎么回应？

`参考回答`

我会强调三点。第一，评测样本要覆盖不同类型查询，不只是标准问句，还要有歧义、长尾和多轮改写。第二，评测集和调参过程要隔离，不能一边看结果一边反复按这批样本去改规则。第三，不能只看一个指标，要结合 badcase 分析，看提升是不是来自真实的长尾问题改善，而不是只把头部样本做得更容易。如果指标提升同时伴随线上 badcase 减少和采纳率提升，那说服力会强很多。

---

## 题 56：准确率和延迟冲突时，你怎么取舍

`模拟问题`

如果 rerank、反思、多步工具调用都能提效果，但会拖慢响应，你怎么平衡？

`参考回答`

我不会默认所有请求都走最重链路，而是按场景分层。简单查询优先走轻链路，复杂任务再走完整链路。对高价值场景，比如测试设计和风险诊断，可以接受更高延迟去换准确性；对低风险信息查询，可以优先保证响应速度。工程上更重要的是做动态路由，而不是为了一两个指标统一把所有流量都压到最重方案上。

---

## 题 57：如果计算预算只够你优化一个点，你先优化哪里

`模拟问题`

预算有限，你只能在检索、模型、工具链路里选一个重点优化，选哪个？

`参考回答`

我大概率优先优化检索和数据质量，因为这是整个系统的上限。很多问题表面上看像模型不够聪明，实际上是上下文错了、数据脏了或者召回不准。尤其在企业场景里，正确上下文往往比更强的生成能力更关键。只有当前置知识链路已经比较稳定之后，再去做 SFT 或 GRPO，投入产出比才更高。

---

## 题 58：你觉得这个项目现在最大的短板是什么

`模拟问题`

如果让你自己批评这个项目，一个最明显的短板是什么？

`参考回答`

我会认为最大的短板通常不是单点能力，而是端到端稳定性还需要更多真实流量验证。离线指标和小范围灰度能证明方向是对的，但真正上线后，输入分布会更杂、使用习惯会更多样、异常链路也会更多。所以我会把这个项目的阶段性结论讲成“已经完成从 demo 到工程系统的关键跨越，但距离大规模稳定使用，还需要继续补灰度验证、权限治理、观测体系和 badcase 闭环”。

---

## 题 59：如果让你重做一次，你会最先改什么

`模拟问题`

回头看这个项目，有没有什么你会更早做的事？

`参考回答`

我会更早把评测体系和 badcase 分类体系建起来。很多团队做 Agent 项目容易一开始就堆 Prompt、堆工具、堆流程，结果线上一出问题很难定位。我现在更认可的顺序是：先定义任务边界和评测口径，再建设数据链路和工具抽象，最后再做模型优化。因为没有评测和 badcase 闭环，后面的优化很容易变成凭感觉调系统。

---

## 题 60：如果再给你三个月，你会怎么继续演进

`模拟问题`

如果这个项目继续做三个月，你的下一步路线图是什么？

`参考回答`

我会优先推进三件事。第一，补线上观测和灰度数据闭环，让采纳率、回退率、人工改写率、异常调用类型能形成稳定看板。第二，继续做 badcase 回流，把检索错误、参数缺失、链路顺序错误这些问题分别沉淀到规则、数据和训练层。第三，推进多 Agent 协作和更细的权限治理，让系统从“能做几个高频任务”升级到“能稳定进入更多测试流程”。换句话说，下一阶段重点不只是继续提模型指标，而是把系统做得更稳、更可控、更能规模化。

## 题 121：DataAgent + RAG 两阶段链路，代码和系统层面是怎么落的

`模拟问题`

你别讲概念，直接讲这条链路在工程里怎么实现。

`参考回答`

我会把它拆成离线链路和在线链路。离线链路先接平台已有文档和附件数据，包括规划文档、脑图、依赖图、历史测试用例、评审意见、测试报告、截图视频等，先做解析和结构化增强，比如 OCR、ASR、chunk 切分和 metadata 建模，再统一入知识库。在线链路则是用户提问后，先做意图识别和 Query Rewrite，再根据项目、版本、模块、任务这些上下文做 metadata 过滤，进入混合检索和重排，最后把高相关证据交给 Agent 去决定是直接回答，还是继续调工具。也就是说，DataAgent 负责把原始数据变成可检索知识对象，RAG 负责把对的问题在对的上下文里召回来。

## 题 122：文档、截图、视频这些不同数据，在入库时怎么统一

`模拟问题`

你们接的东西很杂，最后怎么能进同一套知识链路？

`参考回答`

核心不是让所有数据长得一样，而是让它们都变成统一的“知识对象”。文本类文档重点做分段和结构保留，截图重点做 OCR 并保留来源和时间点，视频重点做关键帧、字幕或 ASR 抽取。最后统一进库的不是原始媒体本身，而是可检索的文本片段、结构化 metadata 和来源引用。这样做的好处是后续检索、过滤、重排和引用逻辑可以统一，不需要为每种数据单独重写一条链路。

## 题 123：chunk 是怎么切的，为什么不是固定长度

`模拟问题`

你们的 chunk 是按多少 token 切的？

`参考回答`

我不会把这个问题答成一个固定数字，因为更关键的是按文档类型和语义边界切，而不是死背一个长度。像版本规划、评审意见、测试报告这种更适合按章节或语义段切；历史用例和缺陷记录更适合按条目、步骤或案例单元切。评估上看的是不同切分策略对 Recall@10、MRR 和 badcase 的影响，而不是某个长度本身多“标准”。如果面试官继续追一个数，我可以补一句，实际工程里会有基础长度和 overlap，但最终不是靠单一长度调出来的。

## 题 124：metadata 到底建了哪些字段，为什么这些字段最重要

`模拟问题`

你总说 metadata 很重要，那你们实际建了哪些字段？

`参考回答`

我会优先讲和业务决策强相关的几类字段：项目、版本规划、研发阶段、里程碑、任务、模块、文档类型、时间范围、来源类型、风险等级。因为你这个场景最大的问题往往不是“没召回”，而是“召回了别的版本、别的模块、别的时间段的内容”。这些字段一旦建好，Query Rewrite 生成的约束条件才能和过滤逻辑对齐，检索空间才真正可控。

## 题 125：为什么你们的 Recall@10 和 MRR 会提升这么多

`模拟问题`

从 `67% -> 89%`、`0.49 -> 0.77`，这个提升主要来自哪里？

`参考回答`

我会把原因拆成三层。第一层是召回层，从单一向量检索升级成 `BGE-M3 + BM25/Elasticsearch` 的混合检索，覆盖了语义匹配和精确实体匹配两类需求。第二层是排序层，引入 `Cross-Encoder Rerank`，把候选里真正最相关的证据推到前面。第三层是约束层，通过 Query Rewrite 和 metadata 过滤减少跨版本、跨模块噪声。简单说，这不是某一个模型突然更强，而是召回、排序、过滤三层一起变得更合理了。

## 题 126：为什么你觉得 Query Rewrite 确实有效，而不是碰巧有效

`模拟问题`

你怎么证明是 Query Rewrite 起作用，不是别的模块顺带带来的？

`参考回答`

比较稳的做法是做分层 ablation，也就是同一批 query 下分别比较“只做向量召回”“加混合检索”“再加 Rewrite”“再加 metadata 过滤”“再加重排”的效果差异。这样能看到 Rewrite 主要改善的是哪类样本，通常是口语化提问、上下文省略和约束不完整的问题。也就是说，我不会只看最终总分上涨，而会看它具体救回了哪些 badcase。

---

## 题 127：Function Calling 这部分，系统上是怎么接的

`模拟问题`

你们说把 35+ 个 Django API 抽象成 Tool Schema，那工程上到底怎么接进去？

`参考回答`

我会把这件事理解成把平台原生后端能力重新包装成模型能稳定消费的动作空间。具体做法是先按业务域拆工具，比如任务、用例、缺陷、质量、报表、附件，再为每个工具定义用途、参数、必填项、枚举范围、错误状态和权限等级。Agent 端不是直接裸调后端接口，而是通过统一 Tool Schema 去做工具选择和参数填充，服务端再做最终鉴权、参数校验和调用。这样模型学的是“业务动作”，不是直接记住某个内部 URL 或字段名。

---

## 题 128：35+ 个工具不会太多吗，模型怎么不混

`模拟问题`

35 个以上的工具，模型不会随便乱选吗？

`参考回答`

工具数量本身不是核心问题，边界是否清楚才是。我的做法是先按域分组，再把高频读工具和高风险写工具区分开，同时在描述里强调使用场景，不只写接口名。真正容易让模型混淆的，不是工具多，而是多个工具做的事高度重叠、描述又不清楚。所以重点不是强行压缩工具数，而是把工具空间做成“可区分、可解释、可约束”的结构。

---

## 题 129：为什么 Prompt + RAG 不够，必须上 SFT

`模拟问题`

你怎么判断这个问题已经不是 Prompt 能解决的，而是要进入 SFT 阶段？

`参考回答`

我会看两个信号。第一，badcase 是否呈现出稳定重复的结构，比如总是选错某几类工具、总是漏某几个关键参数、总是在多步链路里断在相似位置。第二，规则、Prompt、few-shot 和 Tool Schema 调整后，效果是否已经到瓶颈。如果这些低成本手段都用了，问题仍然集中存在，就说明它更像模型动作能力问题，而不是简单提示词问题。这时候再做 SFT 才更有性价比。

---

## 题 130：SFT 数据集到底多大，怎么讲最稳

`模拟问题`

面试官追问你数据集实际是多少条，你怎么答？

`参考回答`

当前最稳的口径，我建议统一讲成：`2k+` Function Calling 样本，其中约 `30%` 是负样本或边界样本，训练集和独立测试集分开。这个说法和资料库里的 Function Calling 项目方法论是一致的，也足够支撑你的简历表达。如果面试官继续追是不是精确到 2173、2384 这种，我不建议硬报一个没有稳定资料支撑的数字，更稳的答法是：这是持续迭代的数据集，不同阶段规模会增长，但对外统一按 `2k+` 去讲。

---

## 题 131：这 2k+ 样本是怎么来的，不会全是 GPT 造的吧

`模拟问题`

你的数据集来源是什么？会不会大部分都是合成垃圾数据？

`参考回答`

我会把来源讲成四类。第一类是平台 API 文档和 Tool Schema，它们定义了动作空间。第二类是历史人工操作链路，也就是测试经理和测试开发真实怎么查、怎么做、怎么汇总。第三类是场景化指令样本，比如版本规划生成任务、结合缺陷趋势判断是否可提测这类问题。第四类才是合成和扩写数据，但这些数据不是直接拿来训练，而是要经过人工校验和规则筛选。也就是说，GPT 可以参与扩写和补齐覆盖，但不是数据质量的最终裁判。

---

## 题 132：负样本为什么要做到 30% 左右

`模拟问题`

你们为什么要放这么多负样本？正样本不够吗？

`参考回答`

因为 Function Calling 场景里最有价值的能力之一，就是“知道什么时候不能这么调”。如果只给模型看正样本，它很容易学会一个错觉，就是任何问题都可以顺利找到工具并调用。但真实线上最容易翻车的，恰恰是歧义项目名、参数缺失、跨版本误召回、工具顺序错误、写工具误触发这些边界场景。所以负样本不是为了凑数据量，而是为了把决策边界学出来。约 `30%` 这个比例比较稳，是因为它既能覆盖边界，又不会把训练集整体分布拉得太偏。

---

## 题 133：训练集、测试集、评测集你们是怎么切的

`模拟问题`

你前面一直说独立测试集，那训练和评测具体怎么隔离？

`参考回答`

我会统一讲成：训练集用于 SFT，独立测试集或独立评测集用于最终离线评估，如果中间有 Prompt、阈值、rerank 策略选择，再额外留验证集。切分上不能只随机切，因为那样很容易让同模板样本同时出现在训练和评测里。更稳的是按时间、场景或者表达模板做隔离，确保评测真的能反映泛化，而不是记住样本形态。

---

## 题 134：工具选择准确率和参数 EM，评测时具体怎么算

`模拟问题`

这两个指标你别泛泛讲，直接讲计算逻辑。

`参考回答`

工具选择准确率看的是：对于当前输入，标准答案里的目标工具有没有被选中。如果是单步任务，就看这一步；如果是多步任务，就拆成单步准确率和链路级成功率。参数 `EM` 看的是结构化参数在归一化后是否和标准答案完全一致，比如项目、版本、模块、任务这些字段都要对上。之所以用 `EM`，是因为对工具调用来说错一个关键字段，结果可能就完全变了，所以它比一般文本相似度更适合这个任务。

---

## 题 135：为什么工具选择准确率能从 79% 提到 92%

`模拟问题`

你觉得这部分提升主要是训练带来的，还是工具设计带来的？

`参考回答`

我不会把功劳都归给训练。更真实的说法是三部分一起起作用。第一，Tool Schema 本身更清晰了，字段和描述更贴近业务动作。第二，数据集覆盖了更多歧义样本和负样本，让模型学到了边界。第三，SFT 让模型在这个固定工具空间里的动作稳定性提高了。所以这个提升不是单纯“模型更强了”，而是工具定义、数据构造和训练一起把动作空间做顺了。

---

## 题 136：参数 EM 为什么能从 71% 提到 86%

`模拟问题`

参数抽取提升这么多，最关键的优化点是什么？

`参考回答`

# Q&A: 指标如何衡量

日期：2026-04-09  
主题：RAG 检索指标、Function Calling 指标、业务效率指标

## 用户问题

最终离线检索上 `Recall@10` 从 `67%` 提升到 `89%`，`MRR` 从 `0.49` 提升到 `0.77`，工具选择准确率从 `79%` 提升到 `92%`，参数 `EM` 从 `71%` 提升到 `86%`，测试用例初稿生成效率提升约 `70%`。  

这几个指标是如何衡量的?

## 回答

如果你要把这几个指标讲稳，建议统一按“同一批离线样本、固定评测口径、基线对比优化版”的方式回答。可以分成三层：

1. 检索层：`Recall@10`、`MRR`
2. 工具调用层：工具选择准确率、参数 `EM`
3. 业务结果层：测试用例初稿生成效率

### 1. 检索层指标怎么衡量

#### 1.1 评测数据怎么来

离线检索评测集建议这样构造：

- 从真实测试场景里抽 query，比如：
  - 根据版本规划生成测试任务
  - 结合历史缺陷判断高风险模块
  - 根据测试报告和评审意见生成风险摘要
- 每条 query 由测试专家或项目成员标注 `gold evidence`
- `gold evidence` 可以是：
  - 规划文档中的正确段落
  - 历史测试用例片段
  - 缺陷趋势相关片段
  - 测试报告 / 评审意见中的关键证据
- 每条 query 可以对应 `1~N` 个正确证据片段

也就是说，检索评测不是看“模型答得像不像”，而是看“正确证据有没有被召回、排得靠不靠前”。

#### 1.2 `Recall@10` 怎么算

标准口径：

`Recall@10 = Top10 中命中的 gold evidence 数 / 该 query 的 gold evidence 总数`

然后对所有 query 取平均。

如果每条 query 只标了 1 个标准证据，那这个指标在实际效果上会接近：

- Top10 里有没有把那条正确证据召回来

所以你在面试里可以这样解释：

`Recall@10 从 67% 到 89%，表示优化后的检索链路在前 10 个候选结果里，更高概率把真正有用的内部知识片段召回来。`

#### 1.3 `MRR` 怎么算

`MRR = 平均值(1 / 第一条正确结果的排名)`

举例：

- 如果第一条正确结果排第 1，得分是 `1`
- 排第 2，得分是 `0.5`
- 排第 5，得分是 `0.2`

所以：

`MRR 从 0.49 到 0.77` 的意思不是“只召回到了更多文档”，而是“正确证据整体排得更靠前了”，这会直接减少后续生成时吃到噪声上下文的概率。

#### 1.4 这个项目里为什么两个指标都要看

- `Recall@10` 解决“有没有召回来”
- `MRR` 解决“排得靠不靠前”

只看 Recall 不够，因为你可能召回到了，但排在第 9、第 10 位，生成时未必真能用上。  
只看 MRR 也不够，因为你可能前几名质量不错，但整体覆盖率不足。

### 2. 工具调用层指标怎么衡量

#### 2.1 评测集怎么来

这部分建议统一说成 `独立测试集` 或 `独立评测集`，来源包括：

- 平台 API 文档和 Tool Schema
- 历史人工操作链路
- 场景化用户指令
- 人工校验后的正负样本

这里“独立测试集 / 评测集”的关键点是：

- 和训练集分开
- 不能把训练里见过的同一批样本原样拿来测
- 最好按场景或时间切分，降低数据泄漏风险

#### 2.2 工具选择准确率怎么定义

标准口径建议这样讲：

`工具选择准确率 = 模型预测的目标工具 与 标注工具 完全一致 的样本数 / 总样本数`

这里默认统计的是：

- 单轮任务的目标工具选择
- 或多步任务里当前 step 的目标工具选择

如果一个任务要调多个工具，建议把“多步链路整体是否成功”放到单独的 `多步任务成功率` 去讲，不要混在工具选择准确率里。

你可以这样解释：

`79% 到 92%` 说明模型在面对平台内多个工具时，更稳定地选对了当前应该调用的那个工具。

#### 2.3 参数 `EM` 怎么定义

这里的 `EM` 是 `Exact Match`，建议口径是：

`参数 EM = 模型输出的结构化参数，在归一化后，与 gold 参数完全一致 的样本数 / 总样本数`

这里的“归一化后”很重要，通常会先统一：

- 项目名别名
- 版本号格式
- 日期格式
- 模块名映射
- 枚举值

只有所有必需字段都对，才算 `EM = 1`。  
只对一部分字段，不算 exact match。

你可以这样讲：

`71% 到 86%` 说明模型不只是知道该调什么工具，而且更稳定地把项目、版本、模块、任务等关键参数抽全、抽准了。

#### 2.4 为什么这两个指标要分开看

因为 Function Calling 常见有两种错法：

1. 工具选错了
2. 工具选对了，但参数抽错了

如果只看最终是否调用成功，会看不清到底是“路由错”还是“参数错”。  
所以工具选择准确率和参数 EM 要拆开统计。

### 3. 业务结果层指标怎么衡量

#### 3.1 “测试用例初稿生成效率提升约 70%”怎么定义

这个指标建议按“从输入需求材料到生成可评审初稿”的耗时来定义。

稳定讲法：

- 起点：测试人员拿到规划文档 / 脑图 / 依赖图 / 历史上下文
- 终点：产出可进入评审流程的首版测试用例初稿
- 对比方式：
  - 人工方式历史平均耗时
  - AI 辅助方式平均耗时

计算方式可以讲成：

`效率提升 = (人工平均耗时 - AI辅助平均耗时) / 人工平均耗时`

例如：

- 人工平均要 `100` 分钟
- Agent 辅助后要 `30` 分钟
- 那就是提升 `70%`

#### 3.2 为什么这是业务指标，不是模型指标

因为它衡量的是：

- 流程效率
- 人机协作价值
- 平台落地效果

不是单纯衡量模型输出质量。

面试时更稳的说法是：

`我们把检索、工具调用、生成和回写串成闭环后，最终关注的不只是离线准确率，还要看真实业务流程里，首版用例草稿能不能更快进入评审。`

### 4. 如果面试官继续追问，你可以这样补

#### 4.1 基线和优化版分别是什么

建议口径：

- 检索层基线：
  - 原始向量检索
  - 或不带 Query Rewrite / Rerank / metadata filter 的版本
- 优化版：
  - `BGE-M3 + BM25/Elasticsearch + Cross-Encoder Rerank`
  - 加上 Query Rewrite 和 metadata 过滤

- 工具调用层基线：
  - Prompt + RAG 直接做工具调用
- 优化版：
  - `Qwen2.5-7B + LoRA` 的 Function Calling SFT 版本

#### 4.2 为什么说这些指标可信

建议口径：

- 使用固定独立测试集，不和训练集混
- 使用统一标注标准
- 基线和优化版在同一批样本上对比
- 检索指标、动作指标、业务指标分层统计

### 5. 面试可直接复用的精简回答

如果面试官直接问“这些指标怎么测的”，你可以直接答：

`我把它分成三层来测。检索层用离线标注的 query-evidence 数据集评 Recall@10 和 MRR，前者看前 10 个候选里有没有把正确证据召回来，后者看第一条正确结果排得靠不靠前。工具调用层用独立测试集评两类指标，一个是工具选择准确率，看工具有没有选对；一个是参数 EM，看项目、版本、模块、任务这些结构化参数在归一化后是否完全匹配。业务层再看测试用例初稿生成效率，也就是从输入规划材料到产出可评审首版用例草稿的耗时，相比人工流程缩短了大约 70%。这样做的好处是能把检索问题、工具问题和业务结果拆开看。`

### 6. 可补充的延伸口径

如果你后面还要讲 GRPO 那组指标，也建议统一成：

- `多步任务成功率`
  - 定义：一个多步任务从检索、工具调用到结果生成是否完整达成目标
- `无效工具调用率`
  - 定义：调用了不该调的工具、重复调工具、参数错导致无效执行等占总调用次数的比例

这样你的整套指标体系会更完整：

1. 检索层：`Recall@10`、`MRR`
2. 调用层：工具选择准确率、参数 `EM`
3. 工作流层：多步任务成功率、无效工具调用率
4. 业务层：用例初稿生成效率、风险摘要产出时长

### 题 10：多工具、多步骤下怎么控制安全性和稳定性

`模拟问题`

既然 Agent 能调很多内部接口，你们怎么避免乱调用，尤其是写操作风险？

`优质回答`

这类平台不能把 Agent 当成完全无约束的执行器，尤其是涉及写接口时。我的做法是先做工具分级，把只读查询工具和有副作用的写工具区分开；再在 Tool Schema 层补充参数约束、幂等要求和错误处理；高风险写操作要么加确认机制，要么限制在白名单场景；结果输出必须尽量带来源引用，方便审计和回放；如果是多步链路，还会在 Reflect 阶段做一次结果审校，避免顺序错、参数错或者上下文不充分时直接写入。简单说，这里不是只追求“能调”，更要保证“调得稳、调得可追溯、调错了能定位”。

### 题 14：你们怎么评估 SFT 的效果

`模拟问题`

你们怎么证明 SFT 有用？只看生成结果吗？

`优质回答`

这类任务不能只看“模型回答看起来不错”，必须看任务级指标。我们主要看三类指标。第一类是工具选择准确率，也就是当前问题有没有选对工具；第二类是参数抽取 `EM`，看结构化参数有没有完整且正确地抽出来；第三类是 `Overall Call Accuracy`，也就是从选工具到参数正确再到最终调用成功的整体准确率。这样做的原因是，Function Calling 的问题通常不是一个文本质量问题，而是一个动作正确性问题。如果工具选错了，回答再通顺也没有价值。如果参数抽错了，系统甚至可能产生错误写入。所以我会强调独立测试集 / 评测集上工具选择准确率 `79% -> 92%`、参数 `EM 71% -> 86%` 这类结果，比单纯展示几个 case 更能说明问题。

### 题 15：为什么还要做 GRPO，而不是停在 SFT

### 题 16：GRPO 的奖励函数怎么设计

`模拟问题`

你们的 GRPO reward 是怎么设计的？为什么这样设计合理？

`优质回答`

我会从任务目标倒推 reward，而不是从算法名词开始讲。对我们的场景来说，最重要的不是把回答写得更像人，而是把多工具工作流走对，所以 reward 会覆盖五类信号。第一类是工具正确性，选没选对工具；第二类是参数正确性，项目、版本、任务、模块这些关键参数抽得对不对；第三类是链路成功率，也就是多步流程最后有没有完成目标任务；第四类是引用可信度，看结果是不是建立在正确的内部知识基础上；第五类是效率惩罚，比如无效调用、重复调用、顺序混乱都应该扣分。这样设计的好处是 reward 直接贴业务目标，不会训练出“看起来会思考，但实际很爱乱调工具”的模型。
